## 2. 操作与API

| 操作名称 | HTTP方法 | API端点 | 请求参数 | 响应格式 | 变更数据 | 刷新范围 |
|---------|---------|---------|---------|---------|---------|---------|
| 获取采集任务列表 | GET | /api/v1/collect-tasks | type (可选): video/transcript/hotspot, keyword (可选), status (可选), page, page_size, sort_by=created_at, order=desc | { items: [{id, task_id, task_type, target_description, target_account_id, status, progress, total_count, success_count, fail_count, duration, created_at}], total } | 无 | 区块D统计、区块E表格 |
| 获取统计摘要 | GET | /api/v1/collect-tasks/stats | — | { today_total, running_count, completed_count, failed_count } | 无 | 区块D统计摘要 |
| 新建采集任务 | POST | /api/v1/collect-tasks | { task_type, target_account_id, date_range, max_count } | { id, task_id, status: 'waiting', ... } | INSERT collect_tasks | 区块D统计、区块E表格（列表顶部插入新行） |
| 立即执行任务 | POST | /api/v1/collect-tasks/{id}/execute | — | { id, status: 'running' } | UPDATE collect_tasks.status→'running' | 区块E对应行（状态变执行中） |
| 暂停任务 | POST | /api/v1/collect-tasks/{id}/pause | — | { id, status: 'paused' } | UPDATE collect_tasks.status→'paused' | 区块E对应行（状态变已暂停） |
| 编辑任务 | PUT | /api/v1/collect-tasks/{id} | { task_type, target_account_id, date_range, max_count } | { id, task_id, ... } | UPDATE collect_tasks | 区块E对应行 |
| 删除任务 | DELETE | /api/v1/collect-tasks/{id} | — | { success: true } | DELETE collect_tasks | 区块D统计、区块E表格 |

| 重新执行 | POST | /api/v1/collect-tasks/{id}/rerun | — | { id, status: 'running' } | UPDATE collect_tasks.status→'running', 重置所有计数 | 区块E对应行（完整重置） |
| 获取采集日志 | GET | /api/v1/collect-tasks/{id}/logs | page, page_size (可选) | { entries: [{timestamp, level, message}], has_more } | 无 | 区块H日志弹窗 |
| 获取定时配置 | GET | /api/v1/collect-tasks/schedule | — | { frequency, execute_time, is_enabled } | 无 | 区块I定时配置弹窗 |
| 保存定时配置 | PUT | /api/v1/collect-tasks/schedule | { frequency, execute_time, is_enabled } | { frequency, execute_time, is_enabled } | UPDATE/INSERT schedule_config | 区块I弹窗关闭 |
| 获取目标账号列表（弹窗下拉） | GET | /api/v1/benchmark-accounts?brief=true | — | { items: [{id, account_name, platform}] } | 无 | 区块G新建弹窗-目标账号下拉 |

## 4. 状态流转

### 采集任务状态流转

```
等待中(waiting) → 执行中(running) → 已完成(completed)
                              ↘ 失败(failed)
执行中(running) → 已暂停(paused) [暂停]
已暂停(paused) → 执行中(running) [恢复]
失败(failed) → 执行中(running) [重试失败项]
已完成(completed) → 执行中(running) [重新执行]
```

允许的转换:
- waiting → running：运营者点击"执行"按钮 或 定时任务自动触发
- running → paused：运营者点击"暂停"按钮
- paused → running：运营者点击"恢复"按钮
- running → completed：采集全部完成（success_count + fail_count = total_count）
- running → failed：采集过程中发生不可恢复错误（如平台反爬）
- failed → running：运营者点击"重试失败项"，仅重试失败的条目
- completed → running：运营者点击"重新执行"，完整重新采集

触发条件:
- 自动触发：定时任务达到执行时间（需 schedule_config.is_enabled=true）
- 手动触发：运营者点击执行/暂停/重试/重新执行按钮

### 操作按钮可用性矩阵

| 任务状态 | 执行 | 暂停 | 恢复 | 编辑 | 删除 | 查看日志 | 重试失败项 | 重新执行 |
| ---------- | ---- | ---- | ---- | ---- | ---- | ------- | --------- | --------- |
| 等待中 | ✅ | — | — | ✅ | ✅ | ✅ | — | — |
| 执行中 | — | ✅ | — | — | — | ✅ | — | — |
| 已暂停 | — | — | ✅ | ✅ | ✅ | ✅ | — | — |
| 已完成 | — | — | — | — | ✅ | ✅ | — | ✅ |
| 失败 | — | — | — | — | ✅ | ✅ | ✅ | — |

## 6. 待确认项

- [ ] **任务ID编号规则**：task_id 格式为"DC-001"，需确认编号生成规则。是自增序列还是按类型前缀+日期+序号？建议使用编号生成服务，前缀 DC (Data Collection)
- [ ] **定时配置存储方式**：schedule_config 是作为 collect_tasks 表的 JSON 字段，还是独立的 schedule_configs 表？如果是全局定时配置（所有任务共用），建议独立表；如果是每个任务独立配置，建议 JSON 字段
- [ ] **duration 计算方式**：collect_tasks 表是否存储 started_at 和 completed_at 字段？还是仅存储 duration 秒数？建议存储 started_at + completed_at，前端按需格式化
- [ ] **日志存储方式**：collect_task_logs 是独立表还是文件存储？实时推送建议使用 SSE (Server-Sent Events)，日志写入独立表 collect_task_logs(task_id, timestamp, level, message)
- [ ] **进度更新方式**：执行中任务的进度更新，是前端轮询(5s)还是后端推送(SSE/WebSocket)？建议 SSE 推送以降低延迟和请求量
- [ ] **批量操作范围**：页面描述提到"全选删除"，是否需要其他批量操作（批量执行、批量暂停）？当前仅设计批量删除
- [ ] **target_description 生成**：是前端拼接（平台+账号名+时间范围），还是后端在创建任务时生成存储？建议后端创建时生成并存储
- [ ] **失败自动重试**：页面描述提到"采集失败自动重试1次后仍失败→待手动处理"，这个自动重试逻辑在哪一层实现？建议在采集引擎层实现，数据库记录重试次数
- [ ] **后台继续执行**：页面描述提到"采集进行中关闭页面→后台继续执行"，这要求采集任务是服务端异步执行的，需确认采集引擎架构

## 7. 对现有系统的影响

- **新增API**:
  - GET /api/v1/collect-tasks — 获取采集任务列表（支持类型筛选+搜索+状态筛选+分页+排序）
  - POST /api/v1/collect-tasks — 新建采集任务
  - POST /api/v1/collect-tasks/{id}/execute — 立即执行任务
  - POST /api/v1/collect-tasks/{id}/pause — 暂停任务
  - PUT /api/v1/collect-tasks/{id} — 编辑任务（仅等待中状态）
  - DELETE /api/v1/collect-tasks/{id} — 删除任务
  - DELETE /api/v1/collect-tasks/batch — 批量删除任务
  - POST /api/v1/collect-tasks/{id}/retry — 重试失败项
  - POST /api/v1/collect-tasks/{id}/rerun — 重新执行
  - GET /api/v1/collect-tasks/{id}/logs — 获取采集日志
  - GET /api/v1/collect-tasks/stats — 获取统计摘要
  - GET /api/v1/collect-tasks/schedule — 获取定时配置
  - PUT /api/v1/collect-tasks/schedule — 保存定时配置
  - GET /api/v1/benchmark-accounts?brief=true — 获取对标账号简要列表（弹窗下拉用）
  - SSE /api/v1/collect-tasks/{id}/progress — 实时进度推送（如采用SSE方案）
- **新增/修改表**:
  - collect_tasks 表：需包含 task_id, task_type, target_account_id, target_description, status, progress, total_count, success_count, fail_count, max_count, date_range(JSON), started_at, completed_at, created_at, updated_at 字段
  - collect_task_logs 表（新建）：task_id(FK→collect_tasks.id, 需索引), timestamp, level, message
  - schedule_configs 表或字段（新建/修改）：frequency, execute_time, is_enabled，需确认存储方式
  - 索引：collect_tasks.status (统计), collect_tasks.task_type (筛选), collect_tasks.created_at (排序), collect_tasks.target_account_id (关联)
  - 外键：collect_tasks.target_account_id → benchmark_accounts.id（需索引）
- **影响的页面**:
  - 仪表盘：今日任务列表可能显示采集任务，对标新增爆款统计依赖采集结果
  - 对标账号管理：立即采集操作触发采集任务创建，采集完成后更新 last_collected_at
  - 爆款视频库：采集完成后可跳转查看，需支持 task_id 和 account_id 筛选
