### 区块F：数据速览

| 页面元素 | 字段名 | 来源类型 | 具体来源 | 备注 |
|---------|--------|---------|---------|------|
| 昨日播放量 | yesterday_play_count | 📊计算 | SUM(video_metrics.play_count) WHERE video_metrics.date=yesterday AND video_metrics.user_id=? | 大数值显示"X.X万" |
| 昨日播放量趋势 | play_count_trend | 📊计算 | (yesterday_play_count - day_before_yesterday_play_count) / day_before_yesterday_play_count * 100 | 保留1位小数，前日为0时显示"--" |
| 对标新增爆款 | new_viral_count | 📊计算 | COUNT(viral_videos) WHERE viral_videos.created_at >= today_start AND viral_videos.user_id=? | 当日新增爆款视频数 |
| 对标新增爆款趋势 | viral_trend | 📊计算 | today_viral_count - yesterday_viral_count | 差值，正数显示↑，负数显示↓ |
| 待确认事项数 | pending_items_count | 📊计算 | COUNT(pending_items) WHERE pending_items.user_id=? AND pending_items.status='pending' | 系统统计待确认项 |
| 待确认事项描述 | pending_description | 📊计算 | 逻辑：pending_items_count > 0 ? "{N}项需处理" : "无待处理" | 动态文案 |
| 互动率 | interaction_rate | 📊计算 | (SUM(like_count) + SUM(comment_count) + SUM(collect_count)) / SUM(play_count) * 100 WHERE video_metrics.date=yesterday AND video_metrics.user_id=? | 保留1位小数，播放量为0显示"--"；单位% |
| 互动率趋势 | interaction_trend | 📊计算 | yesterday_rate - day_before_yesterday_rate | 差值，正数显示↑，负数显示↓ |

## 4. 状态流转

### 任务状态流转

```
待办(todo) → 采集(collecting) → 选题(selecting) → 制作(producing) → 发布(publishing) → 监控(monitoring) → 已完成(completed)
```

允许的转换:
- 待办 → 采集：运营者启动采集任务
- 采集 → 选题：采集数据就绪
- 选题 → 制作：选题确认完成
- 制作 → 发布：视频制作完成
- 发布 → 监控：视频发布成功
- 监控 → 已完成：监控周期结束
- 任意状态 → 待办：运营者手动回退

触发条件:
- 自动推进：上一步骤完成（采集完成/选题确认/视频渲染完成/发布成功/监控周期结束）
- 手动推进：运营者点击"推进到下一步"

### 热点使用状态流转

```
未使用 → 已用于选题 → 已过期
```

允许的转换:
- 未使用 → 已用于选题：运营者在选题工作台使用了该热点
- 未使用 → 已过期：超过 valid_until 时间

## 6. 待确认项

- [ ] **数据速览指标的数据来源**：yesterday_play_count 和 interaction_rate 需要从哪个表聚合？假设存在 video_metrics 表记录每日视频指标，需确认表结构
- [ ] **待确认事项的数据模型**：pending_items 是独立表还是从多个模块聚合（如待确认选题、待审核视频等）？需确认统计逻辑
- [ ] **热点推荐算法**：relevance_score 的计算逻辑是否由 AI 服务异步写入 hotspots 表？前端只读取最终值？
- [ ] **互动率计算**：是否在 video_metrics 表中直接存储每日互动率，还是实时计算？建议存储以提升查询性能
- [ ] **工作流步骤映射**：tasks.current_step 存储的是步骤编号(1-5)还是步骤名称？建议存编号，映射关系前端维护

- [ ] **热点推荐接口**：/hotspots/recommended 是否需要传入人设配置参数用于计算匹配度？还是后端自动关联当前用户的人设？

## 7. 对现有系统的影响

- **新增API**:
  - GET /api/v1/dashboard/workflow — 获取工作流状态
  - GET /api/v1/dashboard/stats — 获取数据速览聚合统计
  - GET /api/v1/hotspots/recommended — 获取推荐热点（按匹配度排序）
  - GET /api/v1/notifications/unread-count — 获取未读通知数
- **新增/修改表**:
  - tasks 表：需确认 current_step 字段类型（建议 SMALLINT，1-5映射）
  - video_metrics 表（可能需新建）：存储每日视频播放量、点赞、评论、收藏等指标，用于计算数据速览
  - notifications 表（可能需新建）：存储系统通知，用于角标提示
  - pending_items 表或视图（可能需新建）：聚合待确认事项
  - hotspots 表：需包含 relevance_score、source_platform、source 字段
- **影响的页面**:
  - 本页面为系统首页，所有其他页面通过 Sidebar/快捷操作入口跳转
  - 选题工作台：接收热点推荐点击传入的 hotspot_id、hotspot_title
  - 各模块页面：接收任务详情点击传入的 task_id、current_step
