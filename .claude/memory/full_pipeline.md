# 完整开发链路

## 阶段总览

```
想法 → 需求 → 命名锚点 → 设计规格 → 页面描述 → 视觉稿 → 数据流 → 技术规格 → 对齐 → 施工图 → 编码 → 验证 → 部署
```

## 各环节详解

### 1. /refine — 需求细化
- **输入**: 模糊想法/功能方向
- **输出**: 功能说明文档（用户故事 + 逐页功能说明 + 边缘场景）
- **核心规则**: 写不出来=没想清楚；核心故事<5个；每步产出文字
- **快捷路径**: 需求已清晰时跳过

### 2. /init-schema — 建立命名锚点
- **输入**: 需求文档
- **输出**: shared-schema.ts 初始版本（枚举 + 实体名 + 字段名）
- **核心规则**: 命名只做一次，后续只能引用不能创造
- **快捷路径**: 已有 schema 时跳过

### 3. /design-spec — 设计规格
- **输入**: 需求文档
- **输出**: 设计规格文档（布局/颜色/字体/间距/组件/状态/响应式）
- **核心规则**: 值用 $变量名 引用禁止硬编码；每值具体到像素；字段全覆盖含三态
- **审查**: /spec-review

### 4. /design-detail — 页面描述
- **输入**: 设计规格文件
- **输出**: 页面描述文档（区块拆解 + 功能映射 + 操作定义 + 边缘场景）
- **核心规则**: 具体到每个元素；功能→元素 100% 覆盖；溯源标记保留
- **审查**: /design-review

### 5. /design-page — 视觉设计
- **输入**: 页面描述文件
- **输出**: shejigao.pen 中的新页面
- **核心规则**: 值用 $变量名；每次 batch_design<=25 操作；先 placeholder 再填充
- **审查**: /check-page

### 6. /data-flow — 数据流设计
- **输入**: 页面路径/设计稿
- **输出**: 数据流文档 + shared-schema.ts 扩展（§4§6）
- **核心规则**: 每字段有来源标注（📥🔗📊⛔）；优先复用已有 API；命名与 schema 一致
- **审查**: /dataflow-review

### 7. /tech-spec — 技术规格
- **输入**: 页面描述 + 设计稿 + 数据流
- **输出**: 技术规格文档（组件层级/Props/API/数据库/状态/路由）+ schema 扩展（§3§5）
- **核心规则**: 与数据流对齐；TypeScript 禁止 any；命名与 schema 一致
- **审查**: /techspec-review

### 8. /align — 对齐理解
- **输入**: 问题/需求描述
- **输出**: 对齐确认 + 需求穷举清单
- **核心规则**: 任务明确时自动跳过；7问对齐 + 需求穷举
- **快捷路径**: 任务非常明确时可跳过

### 9. /plan-first — 施工图+自动施工
- **输入**: 技术规格文档
- **输出**: PLAN.md → 逐任务编码 + 测试 + 提交
- **核心规则**: 严格依赖链分层；自动修复最多 3 次；禁止改测试来通过
- **快捷路径**: <10分钟/1文件/无破坏性 → 直接执行

### 10. 编码阶段
- Schema 增量维护：新命名立即更新 shared-schema.ts
- 改前查规范，改后验证再提交
- 测试驱动：后端 API→集成测试，Store→单元测试，展示组件→类型检查

### 11. /check-page — 页面验证
- **输入**: 页面 URL/路径
- **输出**: 视觉/功能/数据/交互差异报告
- **核心规则**: 双源对照（设计稿 + 数据流）；跨阶段回溯对齐

### 12. /deploy-check — 部署检查
- **输入**: git diff（自动扫描）
- **输出**: 6 项检查通过/失败报告
- **核心规则**: 零 localhost/硬编码端口；桩端点有 markStub；构建零 TS 错误

### 13. /bushu — 部署
- **输入**: 无（或 --skip-frontend / --backend-only）
- **输出**: 部署成功 + 健康检查
- **核心规则**: SQLite WAL 先 checkpoint；SCP 只传 .db 主文件；部署后重启后端

## 一键快捷路径
| 命令 | 场景 |
|------|------|
| `/wf-full [页面名]` | 需求完善时一键全链路 |
| `/wf-new [需求]` | 新功能开发一条龙 |
| `/wf dev [URL]` | 手动开发+自动验证部署 |
| `/wf bug [描述]` | 系统化修 bug |
| `/wf review [范围]` | 全面审查修复 |
| `/wf deploy` | 部署检查+部署 |
| `/wf day` | 收工保存上下文 |
