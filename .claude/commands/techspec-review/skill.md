# 技术规格审查 `/techspec-review`

> **使用场景**：`/tech-spec` 输出技术规格后，检查技术规格与上游文档的对齐度。
> **目标**：确保技术规格是"可编码的蓝图"——字段名对齐、类型完整、组件覆盖全面、无实现盲区。
> **前置**：`/tech-spec` 生成的技术规格文档
> **建议时机**：全部技术规格写完后

## 执行方式（Subagent 模式）

本技能通过 Subagent 执行分析，不占用主对话上下文。

**必须使用 Agent tool** 启动分析代理，传入以下参数：
- subagent_type: "general-purpose"
- prompt: 包含下方完整审查清单

代理在独立上下文中读取技术规格和基准文件，交叉比对，**只返回问题清单摘要给主对话**。

## 用法

```
/techspec-review                  # 审查所有技术规格
/techspec-review 选题工作台        # 审查单个技术规格
```

## 执行步骤

### 第1步：收集文件（10秒）

扫描 `开发文档/05-技术规格/*-技术规格.md`，确定审查范围。
同时读取：
- `开发文档/04-数据流/*-数据流.md`（数据流基准）
- `开发文档/02-设计/*-页面描述.md`（页面描述基准）

### 第2步：五维度审查（60秒）

#### 2.1 数据流对齐度

技术规格的字段名、API端点、类型定义必须与数据流文档严格一致。

**检查规则**：

| 检查项 | 说明 |
|--------|------|
| 字段名一致 | 技术规格中的字段名与数据流溯源表中的字段名完全相同 |
| API端点一致 | 技术规格的API服务函数端点与数据流的API契约表一致 |
| 类型定义一致 | TypeScript接口中的字段类型与数据流的来源类型标注对应 |
| 溯源标记保留 | 📥存储/🔗调用/📊计算/⛔标记贯穿技术规格全文 |
| HTTP方法一致 | 技术规格的API函数HTTP方法与数据流定义一致 |
| 请求参数一致 | API函数的请求参数与数据流的请求体/查询参数一致 |
| 响应类型一致 | API函数的返回类型与数据流的响应字段一致 |

#### 2.2 TypeScript类型完整性

检查技术规格中所有 TypeScript 类型定义：

| 检查项 | 说明 |
|--------|------|
| 无 any | 所有类型定义中没有 `any`，包括嵌套类型 |
| Props有类型 | 每个组件 Props 的每个字段都有明确类型 |
| API有类型 | 每个 API 函数的参数和返回值都有类型 |
| 枚举值完整 | 状态、类型等有限集合使用 enum 或 union type 定义 |
| 可选字段标记 | 可选字段用 `?` 标记，必填字段无 `?` |
| 数组类型明确 | 数组字段定义了元素类型（如 `Array<Item>` 而非 `Array<any>`） |
| null/undefined | 可能为空的字段标注了 `| null` 或 `| undefined` |

#### 2.3 组件层级合理性

从页面描述的区块拆解验证组件树：

| 检查项 | 说明 |
|--------|------|
| 区块覆盖 | 页面描述的每个区块都有对应的组件 |
| 嵌套合理 | 组件嵌套层级与区块层级对应（不超过4层） |
| 复用标注 | 每个组件标注了"新增"或"复用" |
| 通用组件识别 | 多个页面共用的组件被识别为通用组件 |
| Props传递合理 | 父→子的Props传递链不过长（不超过3层透传） |

#### 2.4 数据库与路由完整性

**数据库表结构检查**：

| 检查项 | 说明 |
|--------|------|
| 📥字段覆盖 | 数据流中所有 📥存储 字段都在数据库表中 |
| 外键索引 | 所有外键字段有索引声明 |
| 唯一约束 | 编号/ID字段有 UNIQUE 约束 |
| 时间字段 | 表中有 created_at / updated_at |
| 类型匹配 | 数据库字段类型与 TypeScript 类型匹配（如 VARCHAR↔string） |

**路由配置检查**：

| 检查项 | 说明 |
|--------|------|
| 页面覆盖 | 每个页面都有路由条目 |
| 动态路由 | 详情页有 `:id` 参数 |
| 权限标注 | 需要登录的页面有 auth 标记 |
| 懒加载 | 使用 `import()` 动态导入 |

#### 2.5 跨阶段回溯对齐

从当前阶段（技术规格）向前回溯到数据流→页面描述→设计规格→需求文档，验证端到端一致性。

**检查规则**：

| 检查项 | 说明 |
|--------|------|
| 字段名全链路对齐 | 技术规格中的字段名是否与数据流、页面描述、设计规格、需求文档中的定义一致（如同一个字段不叫 `account_name` 又叫 `accountName`） |
| 枚举值全链路对齐 | 状态、类型、平台等枚举值在五层文档中是否一致（如需求定义"监控中/已暂停"，技术规格TypeScript类型不能变成 `'running' | 'paused'`） |
| 操作有需求依据 | 技术规格中定义的每个API函数能追溯到数据流→页面描述→需求文档的操作定义，无自行增加的API |
| 跨页面参数全链路一致 | A页面跳转B页面时传递的参数名和类型，从技术规格路由定义→数据流API→页面描述→需求文档全链路一致 |

**基准文件**：`开发文档/04-数据流/*-数据流.md` + `开发文档/02-设计/*-页面描述.md` + `开发文档/02-设计/*-设计规格.md` + `开发文档/01-需求/*-需求.md`

## 与其他审查技能的关系

| Skill | 审查对象 | 阶段 |
|-------|---------|------|
| `/project-review` | 项目配置 | 项目初始化后 |
| `/req-review` | 需求文档 | `/refine` 后 |
| `/spec-review` | 设计规格 | `/design-spec` 后 |
| `/design-review` | 页面描述 | `/design-detail` 后 |
| `/check-page` | 前端实现 vs 设计稿 | `/design-page` 后 |
| `/dataflow-review` | 数据流文档 | `/data-flow` 后 |
| `/techspec-review` | 技术规格 | `/tech-spec` 后 |
| `/code-review` | 代码质量 | 编码后 |

## 严重程度定义

| 标记 | 含义 | 示例 |
|------|------|------|
| ❌ 严重 | 会导致编码阶段产生bug或对接失败 | 字段名不一致、类型有any、缺少外键索引 |
| ⚠️ 警告 | 不影响功能但不规范 | 状态类型未用联合类型、组件层级过深、缺少可选标记 |
| 💡 建议 | 可优化项 | 可合并的接口、可抽取的通用组件、可简化的类型定义 |

## 注意事项

- 技术规格审查是编码前最后一道质量门禁，问题在此发现修复成本最低
- 技术规格必须严格对齐数据流——同名不同义是编码阶段最常见的bug源
- TypeScript类型完整性检查是此审查的核心价值——类型错误会在运行时爆发
- 如果某个数据流文档不存在，跳过对应页面的对齐检查并标注
