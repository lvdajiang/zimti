# 一天工作开始 `/wf day-start`

> **使用场景**：早上开始工作，一条命令搞定所有准备工作。
> **目标**：描述问题 → 结构化计划 → 切分支 → 自动执行，不需要部署。

## 用法

```
/wf-day-start                        # 交互式：读取待处理清单，确认后执行
/wf-day-start 审查client, 修复排序bug  # 快速模式：一句话描述 → 自动生成计划
/wf-day-start auto                   # 自动模式：从记忆文件读取所有待处理项
```

---

## 执行步骤

### 第1步：收集任务意图（10秒）

支持三种输入：

**模式A：快速指定**（有参数时）
```
/wf-day-start 审查client, 修复排序bug
```
用户一句话描述，交给第2步 `/plan` 结构化。

**模式B：自动读取**（`auto`）
1. 读取 `session_progress.md` 的"待处理"项
2. 读取 `known_issues.md` 的已知问题
3. 合并为候选任务列表

**模式C：交互式**（无参数）
从记忆文件读取待处理项，展示清单让用户选择。

---

### 第2步：生成结构化计划（调用 `/plan`）

将第1步的任务意图，通过 `/plan` 转为结构化计划：

**输入**：
- 模式A：用户的一句话描述
- 模式B/C：记忆文件中的待处理项列表

**`/plan` 执行内容**：
1. 自动加 `[类型]` 标记（审查/修复/开发/算法/检查等）
2. 定位具体文件/目录范围
3. 排列任务执行顺序（审查在前、修复在后、检查在最后）
4. 输出完整计划文本

**输出示例**：

```
📋 工作计划

[审查] frontend/src/views/client/ 客户端代码质量
[修复] combination_service.py 排序bug
[修复] dispatch_service.py 排序bug
[检查]

确认计划？（回车继续，或编辑修改）
```

**确认点**：用户确认计划后，才进入后续步骤。符合"先描述方法，等待批准"原则。

---

### 第3步：自动生成分支名（5秒）

**基于第2步的结构化计划**生成分支名，格式：`类型-月日-简述`

| 计划中的任务类型 | 分支前缀 | 示例 |
|----------------|---------|------|
| 以 `[审查]` 为主 | `review` | `review-0419-client-admin` |
| 以 `[修复]` 为主 | `fix` | `fix-0419-sort-bug` |
| 以 `[开发]` 为主 | `feat` | `feat-0419-order-page` |
| 以 `[算法]` 为主 | `algo` | `algo-0419-dispatch` |
| 混合类型 | `work` | `work-0419-multi-fix` |

**生成规则**：
1. 从结构化计划中提取主要模块名/功能名
2. 多个同类任务取主要模块名
3. 混合类型用 `work` 前缀
4. 简述不超过20字符，用 `-` 连接
5. 已存在同名分支时追加序号：`fix-0419-sort-bug-2`

**展示确认**：

```
🏷️ 分支信息
分支名：fix-0419-sort-bug
基于：main（当前分支）

确认？
```

---

### 第4步：Git 操作（自动）

**4.1 检查状态**
```bash
git status --short
```

- 无未提交改动 → 直接创建新分支
- 有未提交改动 → 提示用户：
  ```
  ⚠️ 检测到未提交改动（N个文件）
  选项：
  1. stash 保存后切分支（推荐）
  2. 在当前分支继续
  3. 取消
  ```

**4.2 创建并切换分支**
```bash
git checkout -b <分支名>
```

**4.3 推送到远程**（可选，默认不推送，用户要求时才推）

---

### 第5步：执行任务（白天模式）

复用 `/wf-batch` 的执行逻辑，但有以下区别：

| 差异点 | wf-batch | wf-day-start |
|--------|----------|-------------|
| 模型推荐 | 展示推荐表，询问确认 | 跳过，默认 glm-5 |
| 模式判断 | 自动检测 | 强制白天模式 |
| 部署 | 含"部署"关键词才部署 | **永远不部署** |
| 终点 | deploy-check 或 bushu | **永远停在 deploy-check** |

对每个任务：
1. 匹配执行策略（同 wf-batch 的类型映射）
2. 调用对应 skill 执行
3. 任务明确则跳过对齐
4. 遇到阻塞记录原因，跳过继续
5. 每完成一个任务汇报结果

---

### 第6步：汇总报告

```
📋 白天执行报告

分支：fix-0419-sort-bug
执行时间：HH:MM ~ HH:MM

### 结果
- 总计：N项
- ✅ 成功：N项
- ⚠️ 阻塞：N项

### 成功列表
- #1 [任务名]: [一句话结果]

### 阻塞列表
- #2 [任务名]: [原因]

### 涉及文件
- [文件1]: [修改内容]

### 下一步
- 晚上部署：/model glm-5.1 && /wf-batch deploy
- 继续开发：直接在这个分支上工作
```

---

### 第7步：自动保存上下文

执行完毕后自动调用 `/save-ctx`，更新进度记录。

---

## 典型场景

### 场景1：早上处理已知问题
```
/wf-day-start auto
```
读待办清单 → `/plan` 生成计划 → 确认 → 生成分支 → 逐个执行 → 汇总

### 场景2：一句话快速开始
```
/wf-day-start 审查client, 修复排序bug
```
`/plan` 结构化 → 确认计划 → `review-0419-client` 分支 → 执行 → 汇总

### 场景3：晚上部署白天改动
```
/model glm-5.1
/wf-batch deploy
```
切换强模型 → 部署白天累积的改动

## 流程图

```
/wf-day-start
  │
  ├─ 收集任务意图（参数/自动/交互）
  ├─ /plan → 结构化计划（用户确认）
  ├─ 基于计划生成分支名（用户确认）
  ├─ git checkout -b <分支名>
  ├─ 逐个执行任务（白天模式，默认glm-5）
  ├─ 汇总报告
  └─ /save-ctx
```

## 原则

- **先计划后执行**：`/plan` 确认后才开始
- **全自动**：除计划确认和 git 冲突外不中断
- **不部署**：白天模式终点永远是 deploy-check
- **默认 glm-5**：不询问模型切换
- **可中断**：阻塞项记录下来，跳过继续
