# 批量挂机工作流 `/wf-batch`

> **使用场景**：白天开发不部署，晚上一次性部署。或者一次性跑完所有任务。
> **目标**：挂机不干预，早上看结果。

## 用法

```
/wf-batch 开发页面A, 修复bug, 审查代码       # 白天模式：做到 deploy-check 为止，不部署
/wf-batch 开发页面A, 修复bug, 审查代码, 部署 # 全流程：包括部署
/wf-batch deploy                              # 部署模式：只做部署（部署白天累积的改动）
/wf-batch auto                               # 自动模式：从待处理清单中全部执行
/wf-batch                                     # 交互式：列出待处理任务，选择要执行的
```

### 三种模式

| 模式 | 触发条件 | 结束点 | 适用场景 |
|------|---------|--------|---------|
| **开发模式** | 任务列表不含"部署" | deploy-check | 白天开发，本地验证后等晚上部署 |
| **全流程模式** | 任务列表包含"部署" | bushu（完成部署） | 晚上挂机，从头到尾 |
| **部署模式** | 关键词只有"deploy"或"部署" | bushu（完成部署） | 部署白天累积的改动 |

### 典型工作日流程

```
白天：
  /model-pick 开发页面A, 修复bug        → 推荐模型
  /model glm-5                          → 切换模型
  /wf-batch 开发页面A, 修复bug, 审查代码 → 开发模式，做到 deploy-check 为止
  ↓ 本地验证通过，但不部署

晚上：
  /model glm-5.1                        → 部署用强模型
  /wf-batch deploy                      → 部署白天累积的改动
```

## 执行原则

1. **不中断**：遇到问题记录下来，跳过继续下一个，不等待用户确认
2. **自动判断**：每个任务用 `/align` 的判断规则决定是否需要确认（任务明确就跳过）
3. **逐个汇报**：每完成一个任务输出简要结果
4. **模式判断**：根据任务列表是否包含"部署"关键词，自动选择开发模式或全流程模式
5. **开发模式终点**：最后一个任务完成后自动执行 `deploy-check`，但不执行 `bushu`
6. **最终汇总**：全部完成后输出汇总报告

## 执行步骤

### 第1步：模型推荐（5秒）

**在确认任务清单之前，先推荐最合适的模型。**

对任务列表中的每个任务，根据关键词匹配推荐模型（规则同 `/model-pick`）：

| 任务关键词 | 推荐模型 |
|-----------|---------|
| 审查/review/scan/check | glm-4.7 |
| 格式/注释/文案/配置 | glm-4.7 |
| 开发/实现/修复/组件/页面/bug | glm-5 |
| 架构/重构/算法/性能/策略 | glm-5.1 |
| 部署/bushu/上线 | glm-5.1 |

然后输出推荐：

```
🎯 模型推荐

| # | 任务 | 推荐模型 |
|---|------|---------|
| 1 | 审查client | glm-4.7 |
| 2 | 修复问题 | glm-5 |
| 3 | 部署 | glm-5.1 |

当前模型：glm-5
建议切换：/model glm-5.1（取最高需求，保底策略）

如需省钱可分批：
  批1（glm-4.7）：任务1 — 扫描类
  批2（glm-5.1）：任务2,3 — 修改+部署

确认模型后继续？
```

**如果当前模型已经是推荐模型，跳过此步骤，直接进入第2步。**
**如果用户说"不用管模型"，跳过此步骤，直接进入第2步。**

### 第2步：解析任务清单（10秒）

支持三种输入格式，统一解析为任务列表：

**格式A：计划文本**（带 `[类型]` 标记，从 `/plan` 生成）
```
/wf-batch
[审查] client前端代码质量
[修复] combination_service.py 排序bug
[检查]
```

**格式B：逗号分隔简写**（自动映射类型标记）
```
/wf-batch 审查client, 修复排序bug, 检查
```

**格式C：无参数** — 读取记忆文件中的待处理任务：
1. 从 `session_progress.md` 读取"待处理"项
2. 从 `known_issues.md` 读取已知问题
3. 合并为候选任务列表

#### 简写自动映射规则

| 关键词 | 映射为 |
|--------|--------|
| 审查/review | `[审查]` |
| 修复/fix/bug | `[修复]` |
| 开发/dev/页面 | `[开发]` |
| 算法/algo | `[算法]` |
| 部署/deploy/上线 | `[部署]` |
| 检查/check | `[检查]` |
| 性能/perf | `[性能]` |
| 组件/component | `[组件]` |
| 页面page | `[页面]` |

#### `@模型` 强制指定

任何任务可用 `@glm-4.7` / `@glm-5` / `@glm-5.1` 强制指定模型，覆盖自动推荐：
```
[修复] 排序bug @glm-5.1
[部署] @glm-5.1
```

#### 解析后展示确认清单

展示任务清单：

```
📋 批量任务清单（共N项）

| # | 任务 | 类型 | 预估耗时 |
|---|------|------|---------|
| 1 | 审查客户端代码质量 | code-review | 5分钟 |
| 2 | 修复已知排序bug | fix | 10分钟 |
| 3 | 审查管理后台代码质量 | code-review | 5分钟 |

确认执行以上任务？（输入序号筛选，或回车全部执行）
```

### 第2步：逐个执行（核心步骤）

对每个任务，执行以下流程：

#### 2.1 判断任务类型

根据任务描述，自动匹配执行策略：

| 任务描述关键词 | 执行策略 |
|--------------|---------|
| "审查"/"review" | 执行 `/code-review` |
| "修复"/"fix" + 具体问题 | 执行 `/code-fix` |
| "修bug"/"xgbug" | 执行 `/xgbug`（完整6步） |
| "部署"/"deploy" | 执行 `/wf-deploy` |
| "页面"/"page" + URL | 执行 `/wf-dev` |
| "算法"/"algo" | 执行 `/wf-algo` |
| "检查"/"check" | 执行 `/deploy-check` 或 `/perf-check` |

#### 2.2 执行任务

- 调用对应的 skill 执行
- 如果任务明确，跳过对齐，直接执行
- 如果遇到无法自动决定的事项，记录到"阻塞项"，跳过继续下一个

#### 2.3 记录结果

每完成一个任务，记录：

```
✅ 任务 #N 完成：[一句话结果]
```

如果失败或阻塞：

```
⚠️ 任务 #N 阻塞：[原因]（需要人工处理）
```

### 第3步：输出汇总报告

所有任务执行完毕后，输出汇总：

```
📋 批量执行报告

执行时间：[开始时间] ~ [结束时间]

### 结果统计
- 总计：N项
- 成功：N项
- 阻塞：N项

### 成功列表
- #1 [任务名]: [一句话结果]
- #3 [任务名]: [一句话结果]

### 阻塞列表（需要人工处理）
- #2 [任务名]: [阻塞原因]
  - 建议：[处理建议]

### 涉及文件
- [文件1]: [修改内容]
- [文件2]: [修改内容]
```

### 第4步：自动保存上下文

执行完毕后自动调用 `/save-ctx`，更新进度记录，确保下次会话能接续。

---

## 阻塞处理规则

| 阻塞情况 | 处理方式 |
|---------|---------|
| 需要用户确认业务逻辑 | 记录问题，跳过，等用户上线 |
| 遇到技术难题无法自动解决 | 记录问题，跳过 |
| 部署失败 | 停止后续部署任务，继续非部署任务 |
| 前一个任务失败影响后续 | 评估影响，能继续就继续，不能就跳过 |

## 与其他 skill 的关系

| Skill | 关系 |
|-------|------|
| `/wf-batch` | 编排器，调用其他 skill 逐个执行 |
| `/wf` 系列 | 单个任务的执行引擎 |
| `/save-ctx` | 执行完毕后自动保存 |
| `/load-ctx` | 下次会话恢复时读取上次结果 |

## 典型使用场景

### 场景1：晚上挂机审查+修复
```
/wf-batch 审查client, 修复client问题, 审查admin, 部署
```

### 场景2：自动处理已知问题
```
/wf-batch auto
```
从 session_progress.md 和 known_issues.md 中读取待处理项，自动执行。

### 场景3：多个页面批量开发
```
/wf-batch 开发页面A, 开发页面B, 开发页面C
```
