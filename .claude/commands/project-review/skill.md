# 项目配置审查 `/project-review`

> **使用场景**：`/init-project` 完成后、或项目结构发生变更后，检查项目骨架是否完整一致。
> **目标**：确保 CLAUDE.md 配置正确、文档目录齐全、技能卡可用、设计系统同步。
> **前置**：`/init-project` 或手动创建的项目结构
> **建议时机**：项目初始化后 / 新增阶段文档后 / 同步模板后

## 执行方式（Subagent 模式）

本技能通过 Subagent 执行分析，不占用主对话上下文。

**必须使用 Agent tool** 启动分析代理，传入以下参数：
- subagent_type: "general-purpose"
- prompt: 包含下方完整审查清单

代理在独立上下文中读取项目配置文件，交叉比对，**只返回问题清单摘要给主对话**。

## 用法

```
/project-review                  # 审查项目配置
```

## 执行步骤

### 第1步：收集文件（10秒）

读取以下文件和目录结构：

| 收集对象 | 路径 | 提取什么 |
|---------|------|---------|
| 项目配置 | `CLAUDE.md` | 技术栈、文档路径索引、技能列表、日常流程 |
| 文档目录 | `开发文档/` 子目录列表 | 00-09 目录是否齐全 |
| 技能卡 | `.claude/skills/*/skill.md` | 技能名称和描述 |
| 设计系统文档 | `开发文档/03-视觉/设计系统规范.md` | 变量定义 |
| 设计系统文件 | `开发文档/03-视觉/shejigao.pen` | 实际变量（通过 pencil MCP） |

### 第2步：四维度审查（60秒）

#### 2.1 目录结构完整性

检查 `开发文档/` 下是否存在以下10个子目录：

| 编号 | 目录 | 存放内容 |
|------|------|---------|
| 00 | 项目/ | 项目配置 |
| 01 | 需求/ | 需求文档 |
| 02 | 设计/ | 设计规格、页面描述 |
| 03 | 视觉/ | 设计系统、Pencil稿 |
| 04 | 数据流/ | 数据流文档 |
| 05 | 技术规格/ | 技术蓝图 |
| 06 | 架构/ | 架构文档 |
| 07 | 契约/ | 业务契约 |
| 08 | 日志/ | 开发日志 |
| 09 | 归档/ | 历史文件 |

**检查规则**：

| 检查项 | 说明 |
|--------|------|
| 目录存在 | 10个目录是否都存在 |
| 无多余目录 | 不在规范中的目录需标注 |
| .gitkeep存在 | 空目录有 .gitkeep 保持 |

#### 2.2 CLAUDE.md 配置正确性

检查 CLAUDE.md 中的配置项：

| 检查项 | 说明 |
|--------|------|
| 无占位符 | 不应出现 `{项目名称}`、`{日期}`、`{XXX}` 等未替换占位符 |
| 技术栈完整 | 前端+后端+数据库技术栈已填写 |
| 路径索引一致 | 文档路径索引中的路径与实际目录结构匹配 |
| 技能列表完整 | 技能速查中列出的技能都有对应的 skill.md 文件 |
| 日常流程完整 | 完整链路包含所有阶段：refine→design-spec→design-detail→design-page→data-flow→tech-spec→编码→验证→部署 |
| 版本日期 | 版本号和日期已填写 |

#### 2.3 技能卡完整性

检查 `.claude/skills/` 目录：

| 检查项 | 说明 |
|--------|------|
| 核心技能齐全 | refine/design-spec/design-detail/design-page/data-flow/tech-spec/align/plan-first 都存在 |
| 审查技能齐全 | spec-review/design-review/check-page/code-review/project-review/req-review/dataflow-review/techspec-review 都存在 |
| skill.md 格式 | 每个技能目录下都有 skill.md 且含 frontmatter |
| 无重复技能 | 没有功能重复的技能卡 |

#### 2.4 设计系统同步状态

比对 `设计系统规范.md` 与 `shejigao.pen` 的变量：

| 检查项 | 说明 |
|--------|------|
| 变量数量一致 | 文档中列出的变量数与 .pen 文件中的变量数一致 |
| 变量值一致 | 文档中的变量值与 .pen 文件中的值匹配 |
| 无孤立变量 | .pen 文件中有但文档中没列出的变量 |
| 无文档幽灵变量 | 文档中列出但 .pen 文件中不存在的变量 |

## 与其他审查技能的关系

| Skill | 审查对象 | 阶段 |
|-------|---------|------|
| `/project-review` | 项目配置 | 项目初始化后 |
| `/req-review` | 需求文档 | `/refine` 后 |
| `/spec-review` | 设计规格 | `/design-spec` 后 |
| `/design-review` | 页面描述 | `/design-detail` 后 |
| `/check-page` | 前端实现 vs 设计稿 | `/design-page` 后 |
| `/dataflow-review` | 数据流文档 | `/data-flow` 后 |
| `/techspec-review` | 技术规格 | `/tech-spec` 后 |
| `/code-review` | 代码质量 | 编码后 |

## 严重程度定义

| 标记 | 含义 | 示例 |
|------|------|------|
| ❌ 严重 | 会导致后续步骤无法进行 | 缺少核心目录、技术栈未填写、核心技能缺失 |
| ⚠️ 警告 | 不影响功能但不规范 | 技能列表缺项、.gitkeep缺失、文档日期过时 |
| 💡 建议 | 可优化项 | 目录命名可更规范、CLAUDE.md描述可更精确 |

## 注意事项

- 如果 shejigao.pen 不存在，跳过2.4维度并标注
- 项目配置审查是唯一不需要跨文档交叉比对的审查，主要是自洽性检查
- 建议在项目初始化后立即执行一次，之后每次项目结构变更后重新执行
