# 根因分析 `/retro`

> **使用场景**：审查发现 ≥ 3 个问题后调用，或问题反复出现时主动调用。
> **目标**：从问题清单中识别模式，定位根因，提出开发体系升级方案。

## 执行方式（Subagent 模式）

**必须使用 Agent tool** 启动分析代理，传入以下参数：
- subagent_type: "general-purpose"
- prompt: 包含下方完整分析步骤 + 问题来源说明

---

## 用法

```
/retro                     # 分析累积问题文件 .claude/review-issues.log
/retro code-review         # 只分析最近的 code-review 结果
/retro check-page          # 只分析最近的 check-page 结果
```

---

## 执行步骤

### 第1步：读取累积问题（10秒）

读取 `.claude/review-issues.log` 文件（各审查技能卡自动追加）。

如果文件不存在或问题总数 < 3，输出：

```
✅ 累积问题不足 3 个（当前 N 个），无需根因分析。
```

结束流程。

### 第2步：模式识别（30秒）

将所有问题按以下维度聚类，找出 ≥ 2 个问题的模式：

| 聚类维度 | 示例 |
|---------|------|
| 问题类别 | 错误处理、类型安全、假操作、分页缺失 |
| 涉及文件 | 同一文件反复出问题 → 该文件设计有缺陷 |
| 问题模式 | N 处 catch 无反馈、N 处裸 Number()、N 处桩未标记 |
| 代码区域 | 全是前端、全是 routes、全是 Vue 文件 |

输出聚类结果：

```
📊 模式识别
模式1: [N] 个同类问题 — [描述]
  - 问题 a
  - 问题 b
模式2: [N] 个同类问题 — [描述]
  - 问题 c
  - 问题 d
```

### 第3步：根因定位（每个模式逐一分析）

对每个模式，用"五问法"向下追问，定位根因：

```
问题 → 为什么会出现 → 直接原因 → 为什么直接原因存在 → 根因
```

**根因分类**：

| 根因类型 | 含义 | 典型表现 | 升级方向 |
|---------|------|---------|---------|
| **缺基础设施** | 项目缺少某个基础能力 | 无 toast 库、无统一错误处理、无桩标记机制 | 引入库/中间件 |
| **技能卡遗漏** | 审查技能卡缺少某类检查项 | code-review 不检查编码模式、check-page 不检查空状态 | 扩展技能卡 |
| **检查规则缺失** | schema-check/lint 没覆盖该模式 | 无裸 Number() 检测、无 catch 静默检测 | 扩展检查脚本 |
| **流程断点** | 开发流程缺少必要环节 | 无桩清单跟踪、无空状态设计前置 | 新增流程步骤 |
| **文档不同步** | shared-schema/设计文档与代码不一致 | 枚举值缺失、API 路径未定义 | 更新文档 |
| **非体系问题** | 单纯的编码疏忽，无需升级体系 | 偶发 typo、单处逻辑错误 | 直接修复 |

输出：

```
🔍 根因分析
模式1 → 根因: [类型] [具体原因]
模式2 → 根因: [类型] [具体原因]
模式3 → 根因: 非体系问题，直接修复即可
```

### 第4步：生成升级方案（只针对体系性根因）

对每个"非体系问题"之外的根因，输出具体的升级方案：

```markdown
## 升级方案

### 升级1: [简短标题]
- **根因**: [类型] [具体原因]
- **改动文件**: [具体文件列表]
- **改动内容**: [具体要做什么]
- **预期效果**: [防住哪些问题]

### 升级2: [简短标题]
- ...
```

**规则**：
- 升级方案必须具体到"改哪个文件的哪个部分"，不能是"加强审查"这种空话
- 优先选择自动化的方案（脚本规则 > 技能卡 > 流程规范）
- 每个升级方案必须关联到它能防住的至少 1 个具体问题

### 第5步：输出报告

```
🔍 根因分析报告

累积问题: N 个（来自 M 次审查）

### 模式与根因

| # | 模式 | 问题数 | 根因类型 | 根因 |
|---|------|--------|---------|------|
| 1 | xxx | N | 缺基础设施 | xxx |
| 2 | xxx | N | 技能卡遗漏 | xxx |
| 3 | xxx | N | 非体系问题 | — |

### 升级方案

| # | 改动 | 文件 | 防住 |
|---|------|------|------|
| 1 | [简述] | [文件] | [问题列表] |

### 直接修复（非体系问题）

| # | 问题 | 修复方式 |
|---|------|---------|
| 1 | xxx | xxx |

下一步: 确认升级方案后执行，完成后清空 .claude/review-issues.log
```

### 第6步：清理（执行升级后）

升级方案执行完毕后，清空累积文件：
- 将 `.claude/review-issues.log` 内容归档（如有价值）
- 或直接清空，等待下一轮累积

---

## 与审查技能卡的集成

以下技能卡在输出报告后，会自动将问题追加到 `.claude/review-issues.log`：

- `/code-review` — 代码审查
- `/check-page` — 页面检查
- `/deploy-check` — 部署检查

追加格式：
```
## [日期 时间] /技能卡名称 审查范围
| # | 严重程度 | 类别 | 文件:行号 | 问题描述 |
|---|---------|------|----------|---------|
| 1 | ❌高 | xxx | xxx:L42 | xxx |
```

---

## 注意事项

- 累积问题 < 3 个时不执行分析，直接修复即可
- "非体系问题"不要强行归类，避免过度设计
- 升级方案优先自动化（脚本/规则），其次人工流程（技能卡/规范）
- 升级完成后必须清空累积文件，否则下一轮会重复分析
