# 回归检查 `/regression`

> **使用场景**：修改了公共组件、工具函数、API、数据库模型后，确认没有破坏其他页面。
> **目标**：修 A 不坏 B，改动后快速验证关联部分。

## 执行方式（Subagent 模式）

本技能通过 Subagent 执行分析，不占用主对话上下文。

**必须使用 Agent tool** 启动分析代理，传入以下参数：
- subagent_type: "general-purpose"
- prompt: 包含下方检查步骤 + 改动文件路径

代理在独立上下文中读取关联代码、分析影响范围，**只返回影响范围摘要给主对话**。禁止在主窗口中读取和分析代码。

当用户调用 `/regression` 时，按以下步骤执行。

---

## 步骤1：分析改动范围

从最近的 git diff 或用户描述中，确定改动类型：

| 改动类型 | 影响范围 | 检查重点 |
|:---------|:---------|:---------|
| 公共工具函数（utils/） | 所有引用该函数的组件 | grep 查找所有引用，逐个确认 |
| 公共组件（common/） | 所有使用该组件的页面 | grep 查找所有 import |
| API 端点修改 | 所有调用该 API 的前端组件 | grep 查找 API 函数名 |
| 数据库模型修改 | 所有使用该模型的 API 端点 | grep 查找模型引用 |
| Schema 修改 | 前端类型定义 + API 调用 | 检查 TypeScript 类型是否匹配 |
| 路由修改 | 导航菜单、权限守卫 | 检查路由引用 |
| 后端服务修改 | 调用该服务的所有 API | grep 查找服务引用 |

## 步骤2：定位关联文件

根据改动类型，用 grep 查找所有引用：

```bash
# 工具函数被谁引用
grep -r "from.*utils/xxx" frontend/src/

# 公共组件被谁使用
grep -r "import.*TagSelector\|import.*TagDisplay" frontend/src/

# API 被谁调用
grep -r "xxxApi\|xxxFunction" frontend/src/

# 模型被谁引用
grep -r "from.*models.*xxx" backend/
```

## 步骤3：逐个检查关联文件

对每个关联文件，检查：

1. **导入是否兼容**：函数签名变了，调用方是否需要更新？
2. **类型是否匹配**：Schema 字段类型/名称变了，前端 TypeScript 是否同步？
3. **逻辑是否正确**：公共函数的行为变化是否影响调用方的预期？
4. **渲染是否正常**：公共组件的 props/事件变了，使用方是否适配？

## 与 /xgbug 的区别

| | /regression | /xgbug |
|:--|:-----------|:-------|
| 触发时机 | 改完代码后主动检查 | 用户报告 bug 后修复 |
| 检查范围 | 按引用关系追踪关联文件 | 按组件类型找同类组件 |
| 目标 | 预防回归（修 A 坏 B） | 根除同类 bug（修一个灭一类） |
| 输出 | 影响范围报告 | 修复报告 |

**最佳配合**：`/regression`（预防）+ `/xgbug`（修复），先防后治。

---

## 注意事项

- 不实际运行页面，只做代码级别的静态分析
- 如果改动范围很大（超过 20 个关联文件），建议用子代理并行检查
- 报告中标记"⚠️ 检查"的文件，建议用户手动在浏览器验证
