# 数据流审查 `/dataflow-review`

> **使用场景**：`/data-flow` 输出数据流文档后，检查数据流设计质量。
> **目标**：确保字段溯源完整无遗漏、API设计跨页面一致、与页面描述的映射无缺口。
> **前置**：`/data-flow` 生成的数据流文档
> **建议时机**：全部页面数据流写完后

## 执行方式（Subagent 模式）

本技能通过 Subagent 执行分析，不占用主对话上下文。

**必须使用 Agent tool** 启动分析代理，传入以下参数：
- subagent_type: "general-purpose"
- prompt: 包含下方完整审查清单

代理在独立上下文中读取所有数据流文档和基准文件，交叉比对，**只返回问题清单摘要给主对话**。

## 用法

```
/dataflow-review                  # 审查所有数据流文档
/dataflow-review 选题工作台        # 审查单个数据流文档
```

## 执行步骤

### 第1步：收集文件（10秒）

扫描 `开发文档/04-数据流/*-数据流.md`，确定审查范围。
同时读取：
- `开发文档/02-设计/*-页面描述.md`（字段基准）
- `开发文档/06-架构/architecture.md`（架构基准，如有）

### 第2步：五维度审查（60秒）

#### 2.1 字段溯源完整性

从页面描述的区块拆解中提取所有 📥存储/🔗调用/📊计算 字段，与数据流文档的字段溯源表逐项比对。

**检查规则**：

| 检查项 | 说明 |
|--------|------|
| 页面字段全覆盖 | 页面描述中的每个数据绑定字段在数据流中有溯源 |
| 溯源类型正确 | 📥存储→数据库字段 / 🔗调用→关联查询 / 📊计算→计算公式 / ⛔禁止→说明原因 |
| 无"待定"来源 | 不允许出现数据来源为"待定"的字段 |
| 数据库表存在 | 📥存储字段引用的表和字段在数据库表结构中存在 |
| 外键关系正确 | 🔗调用字段的外键路径可走通 |
| 计算公式明确 | 📊计算字段有明确的计算逻辑描述 |
| 无禁止字段泄露 | ⛔禁止标记的字段不出现在前端类型定义中 |

#### 2.2 API跨页面一致性

跨所有数据流文档，检查同一资源的API设计：

| 检查项 | 说明 |
|--------|------|
| 端点一致 | 同一资源的列表/详情API在不同页面中端点相同 |
| 方法一致 | 同一操作（如删除）在不同页面中HTTP方法相同 |
| 参数名一致 | 同一字段在不同API中的参数名相同（如都用 `page_size` 而非混用 `pageSize`） |
| 响应结构一致 | 同一资源在不同API中的响应字段结构一致 |
| 变更数据声明 | 写操作（POST/PATCH/DELETE）声明了影响的数据表 |
| 刷新范围合理 | 操作后的刷新范围覆盖了所有受影响的UI区块 |

#### 2.3 页面描述映射完整性

**检查规则**：

| 检查项 | 说明 |
|--------|------|
| 显示字段→API | 页面描述的每个显示字段在数据流中有对应的API获取方式 |
| 操作→API | 页面描述的每个操作在数据流中有对应的API端点 |
| 页面状态→API | 加载状态、空状态、错误状态的数据获取方式有说明 |
| 参数传递匹配 | 页面描述的跨页面参数在数据流中有对应的API参数传递 |
| 筛选→查询参数 | 页面描述的筛选控件在数据流中有对应的查询参数 |
| 分页→API参数 | 分页控件在API中有 page/pageSize/total 支持 |

#### 2.4 状态流转合理性

检查数据流文档中的状态流转图：

| 检查项 | 说明 |
|--------|------|
| 状态完整 | 页面描述中定义的所有状态值都在流转图中 |
| 转换有触发 | 每个状态转换有对应的操作或事件触发 |
| 转换有API | 每个状态转换有对应的API调用 |
| 无孤立状态 | 没有无法到达的状态 |
| 无死锁状态 | 没有无法离开的状态（终态除外） |
| 前端状态归属 | 数据流文档声明了每块数据的归属（组件内/Pinia/URL） |

#### 2.5 跨阶段回溯对齐

从当前阶段（数据流）向前回溯到页面描述→设计规格→需求文档，验证端到端一致性。

**检查规则**：

| 检查项 | 说明 |
|--------|------|
| 字段名对齐 | 数据流中的每个字段名是否与页面描述、设计规格和需求文档中的定义一致（如同一个字段不叫 `account_name` 又叫 `accountName`） |
| 枚举值对齐 | 状态、类型、平台等枚举值在四层文档中是否一致（如需求定义"监控中/已暂停"，数据流不能变成"运行中/已停止"） |
| 操作有需求依据 | 数据流中定义的每个API操作能追溯到需求文档的用户故事或操作定义，无自行增加的API |
| 跨页面参数双向对称 | A页面传出参数给B，B的API入参定义必须匹配（名称+类型），且与页面描述中的跨页面关联一致 |

**基准文件**：`开发文档/02-设计/*-页面描述.md` + `开发文档/02-设计/*-设计规格.md` + `开发文档/01-需求/*-需求.md`

## 与其他审查技能的关系

| Skill | 审查对象 | 阶段 |
|-------|---------|------|
| `/project-review` | 项目配置 | 项目初始化后 |
| `/req-review` | 需求文档 | `/refine` 后 |
| `/spec-review` | 设计规格 | `/design-spec` 后 |
| `/design-review` | 页面描述 | `/design-detail` 后 |
| `/check-page` | 前端实现 vs 设计稿 | `/design-page` 后 |
| `/dataflow-review` | 数据流文档 | `/data-flow` 后 |
| `/techspec-review` | 技术规格 | `/tech-spec` 后 |
| `/code-review` | 代码质量 | 编码后 |

## 严重程度定义

| 标记 | 含义 | 示例 |
|------|------|------|
| ❌ 严重 | 会导致编码阶段对接出错 | 字段无溯源、API端点冲突、操作无API对应 |
| ⚠️ 警告 | 不影响功能但不规范 | 参数命名风格不统一、计算公式描述不够精确 |
| 💡 建议 | 可优化项 | 可合并的API、可简化的查询、可缓存的字段 |

## 注意事项

- 数据流审查必须在所有相关页面描述完成后执行，跨页面一致性需要全量比对
- 如果架构文档不存在，跳过数据库表存在性检查并标注
- API一致性检查遵循 REST 规范，同一资源的 CRUD 应使用统一前缀
- 前端状态归属检查是数据流文档的必要部分，缺失则标记为警告
