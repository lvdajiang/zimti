# 模型选择 `/model-pick`

> **使用场景**：开始任务前，不知道该用哪个模型。
> **目标**：一句话推荐，省钱不浪费。

## 用法

```
/model-pick                    # 交互式：描述任务后推荐
/model-pick 修bug               # 快速推荐
/model-pick 审查client, 修复, 部署  # 批量任务推荐
```

## 可用模型

| 模型 | 定位 | 适用场景 |
|------|------|---------|
| glm-4.7 | 轻量快速 | 简单扫描、格式调整、文档、小改动 |
| glm-5 | 均衡 | 页面开发、代码审查、bug修复、常规开发 |
| glm-5.1 | 最强 | 架构设计、算法修改、复杂重构、部署 |

## 选择规则

### 单任务

| 任务类型 | 关键词 | 推荐模型 |
|---------|--------|---------|
| 扫描/审查 | 审查、review、scan、check | glm-4.7 |
| 简单修改 | 格式、注释、文案、配置 | glm-4.7 |
| 常规开发 | 开发、实现、修复、组件、页面 | glm-5 |
| bug修复 | bug、xgbug、错误、报错 | glm-5 |
| 复杂任务 | 架构、重构、算法、性能优化、策略 | glm-5.1 |
| 部署 | 部署、bushu、上线 | glm-5.1 |
| 批量任务 | 多个任务混合 | 取最高需求（保底策略） |

### 批量任务

**原则：取任务列表中需求最高的模型，用它跑全部。**

因为会话中途无法切换模型，必须用一个模型跑完全部任务。选择最强的那个确保不出错。

如果用户想省钱，可以建议分批：

```
📋 批量任务模型推荐

任务1: 审查client → glm-4.7
任务2: 修复问题   → glm-5
任务3: 部署       → glm-5.1

建议方案：
方案A（一次跑完）: /model glm-5.1 → 全部用 glm-5.1（保底，最安全）
方案B（分两批）:
  批1: /model glm-4.7 → 审查client（省资源）
  批2: /model glm-5.1 → 修复 + 部署（确保质量）
```

## 执行步骤

### 第1步：分析任务（5秒）

根据用户输入的关键词，匹配任务类型和推荐模型。

### 第2步：输出推荐

**单任务：**
```
🎯 推荐模型：glm-5

原因：bug修复属于中等复杂度任务，glm-5 的能力足够。

切换命令：/model glm-5
```

**批量任务：**
```
🎯 批量任务模型推荐

| # | 任务 | 推荐模型 |
|---|------|---------|
| 1 | 审查client | glm-4.7 |
| 2 | 修复问题 | glm-5 |
| 3 | 部署 | glm-5.1 |

建议：/model glm-5.1（取最高需求，保底策略）
```

### 第3步：等待用户切换

用户执行 `/model xxx` 后，即可开始工作。

## 注意事项

- 模型选择建议仅供参考，用户可自行决定
- 如果用户已经选了模型，不需要再次推荐
- 部署类任务建议用 glm-5.1，部署出错代价高
