# 新功能开发工作流 `/wf-new`

> **使用场景**：开发一个新功能、新页面、新模块时。
> **目标**：一条命令走完全流程，从需求确认到生产部署。编码阶段全程自动推进，仅在决策点暂停。

## 用法

```
/wf-new <需求描述>                    # 一条龙（对齐+施工图+自动施工+检查+部署）
/wf-new admin新增财务管理报表            # 带具体描述
/wf-new 客户端增加订单导出PDF功能      # 带具体描述
/wf-new --plan-only                     # 只生成施工图不执行（评审用）
```

---

## 流程图

```
/wf-new:
  align → plan-first(施工图+自动施工) → check-page → perf-check → component-check → deploy-check → bushu
```

---

## 步骤1：需求对齐（强制，不可跳过）

**技能**: `/align` 自动执行

**每次都必须执行**，通过问答明确需求后再进入计划阶段。

执行流程：
1. 读取用户的需求描述作为初始输入
2. 执行 `/align` 7问对齐，逐项确认：
   - 功能边界（做什么/不做什么）
   - 涉及模块（哪些页面/组件）
   - 数据来源（哪些表/接口）
   - 展示要求（列表/表单/弹窗）
   - 业务规则（校验/权限/状态流转）
   - 优先级与约束（必须/可选/截止时间）
   - 验收标准（怎样算完成）
3. 对齐结果确认后，进入步骤2

**原则**: 宁可多问一句，不要返工一次。

---

## 步骤2：施工图 + 自动施工

**技能**: `/plan-first`（施工图版）

本步骤是核心改变——不再使用 `/plan` + `/wf-batch` 的扁平任务模式，改为 `/plan-first` 的结构化施工图 + 自动施工模式。

### 2.1 生成施工图

`/plan-first` Phase 1 自动执行：
1. 读取技术规格文档（如果存在）或根据对齐结果分析
2. 依赖分析：自动排列任务执行顺序（DB迁移→后端路由→共享类型→前端API层→Store→组件→页面→联调）
3. 生成 PLAN.md，包含：
   - 分阶段任务列表（每个任务标注文件、依赖、完成标准）
   - 验证节点（阶段级验证）
   - 决策点（`[决策]` 标记需人工判断的任务）
4. **STOP**：展示施工图，等待用户 `[Approved]`

### 2.2 自动施工

用户输入 `[Approved]` 后，`/plan-first` Phase 2 自动执行：

```
FOR each 阶段:
  FOR each 任务（按依赖顺序）:
    执行 → 验证完成标准
    失败 → 自动修复（最多3次）→ 再验证
    仍失败 → 暂停，汇报问题
  阶段完成 → 执行验证节点 → 汇报进度
全部完成 → 汇报
```

**仅在以下情况暂停**：
- 自动修复3次仍失败
- 遇到 `[决策]` 标记
- Tech Spec 有矛盾或遗漏
- 发现影响已有功能的风险

**汇报频率**：每完成一个阶段汇报一次，暂停时立即汇报。

**提交策略**：每个阶段完成后自动 `git commit`。

详见 `/plan-first` 技能卡。

## 步骤3：验证与部署

施工图全部完成后，依次执行：

```
步骤3.1: /check-page      → 对照设计稿检查（有前端页面时）
步骤3.2: /perf-check      → 性能预算检查（列表页自动）
步骤3.3: /component-check → 组件统一性检查
步骤3.4: /deploy-check    → 部署前检查
步骤3.5: /bushu           → 部署到服务器
```

## 与现有工作流的关系

| 工作流 | 关系 |
|-------|------|
| `/wf-new` | **编排器**，串联全流程 |
| `/plan-first` | 编码阶段的施工图生成器 + 自动施工引擎（步骤2） |
| `/wf-dev` | 手动开发模式（适用于没有 tech-spec 的快速开发） |
| `/plan` + `/wf-batch` | 批量多页面开发模式（与 `/wf-new` 独立） |
| `/align` | 需求阶段自动调用（步骤1） |
| `/check-page` | 验证阶段（步骤3） |
| `/save-ctx` | 部署完成后自动保存上下文 |

## 示例

### 示例1：有技术规格的新页面

```
用户: /wf-new 仪表盘（任务中心）

自动执行流程：
1. /align → 7问快速确认
2. /plan-first →
   读取 仪表盘-技术规格.md
   生成施工图（4个阶段，15个任务）
   用户确认 [Approved]
3. 自动施工 →
   阶段一：DB迁移 + API端点 ✅
   阶段二：路由 + API层 + Store ✅
   阶段三：15个Vue组件 ✅
   阶段四：页面联调 ✅
4. /check-page → /perf-check → /bushu
```

### 示例2：明确需求的小功能

```
用户: /wf-new 在仪表盘添加数据导出按钮

自动执行流程：
1. /align → 快速确认（导出格式？数据范围？触发方式？）
2. /plan-first →
   分析：小功能，2个阶段
   阶段一：后端导出API
   阶段二：前端按钮+调用
   用户确认 [Approved]
3. 自动施工 → 全部通过
4. /check-page → /bushu
```

## 原则

- **需求优先**：先确认做什么，再想怎么做
- **双源对照**：前端开发必须对照 shejigao.pen + 数据逻辑文档
- **施工图驱动**：编码前必须生成施工图，禁止凭经验直接写
- **自动推进**：施工图批准后全程自动，仅在决策点和失败处暂停
- **分阶段提交**：每个阶段完成后自动 commit，降低回滚风险
- **验证不跳过**：check-page、perf-check、deploy-check 不可跳过
