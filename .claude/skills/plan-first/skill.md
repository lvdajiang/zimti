# 规划-执行分离模式（施工图版）

> **核心思想**: Tech Spec 是蓝图（描述最终状态），本技能将其翻译为施工图（描述每步操作），然后自动施工。
> **目标**: 编码阶段全程自动推进，仅在决策点和无法自动修复的问题处暂停。
> **质量保障**: 每个任务自动生成测试 → 跑测试 → 红→修→绿→提交，用测试替代人工审读代码。

---

## Phase 1: 调研与生成施工图

### 前置读取（如果存在）

- `开发文档/05-技术规格/{页面名}-技术规格.md` → 组件层级、Props、API、表结构
- `开发文档/04-数据流/{页面名}-数据流.md` → API契约、字段溯源
- 当前代码库中已有的相关文件

### 第1步：依赖分析

从技术规格中提取所有需要创建/修改的文件，按标准依赖链排列执行顺序：

```
标准依赖链（不可逆）：

Layer 0 - 数据层:    Prisma 模型 + 迁移脚本
Layer 1 - 后端层:    路由 + Service + Controller
Layer 1T- 后端测试:  API 集成测试（vitest + supertest）
Layer 2 - 共享类型:   TypeScript 接口（前后端共用）
Layer 3 - 前端API层:  api/*.ts（调用后端）
Layer 4 - 状态管理:   Pinia Store + Store 单元测试
Layer 5 - 叶组件:     无子组件依赖的基础组件
Layer 6 - 组合组件:   由叶组件组装的容器组件
Layer 7 - 页面组装:   路由配置 + 页面入口
Layer 8 - 联调验证:   全页面联调 + /check-page + 全量测试通过
```

**并行化规则**：
- Layer 0-1 之间顺序执行
- Layer 2 可与 Layer 0-1 并行（纯类型定义，不依赖运行时）
- Layer 5 内的叶组件之间可并行
- Layer 6 内的组合组件之间可并行（如果它们不互相引用）
- Layer 7 依赖 Layer 6 全部完成

### 第2步：生成施工图

创建 `PLAN.md`，严格按以下格式：

```markdown
# 施工图：{页面名}

## 目标
{一句话描述本次要完成的功能}

---

## 执行计划

### 阶段一：后端基础
- [ ] 1.1 {任务描述}
  - 文件: {file1.ts, file2.ts}
  - 依赖: 无
  - 完成标准: {可验证的具体条件，如 "npx prisma migrate dev 成功，schema-check 通过"}
  - 测试: {测试描述，如 "prisma migrate 成功，seed 数据可查询"}

- [ ] 1.2 {任务描述}
  - 文件: {file.ts}
  - 依赖: 1.1
  - 完成标准: {如 "curl /api/v1/xxx 返回 200"}
  - 测试: {如 "POST /api/v1/xxx 参数校验 → 201，缺少必填 → 400，无权限 → 403"}

- [ ] 1.T1 {后端 API 集成测试}

### 阶段二：前端骨架
- [ ] 2.1 {任务描述}
  - 文件: {file.ts}
  - 依赖: 1.2（可与阶段一后半并行）
  - 完成标准: {如 "访问 /xxx 显示页面框架，控制台无报错"}
  - 测试: {如 "类型检查通过，API 模块导出正确"}

- [ ] 2.2 {任务描述}
  - 文件: {file.ts}
  - 依赖: 2.1
  - 完成标准: {如 "TypeScript 编译通过，类型与 shared-schema.ts 对齐"}
  - 测试: {如 "store 初始状态正确，action 调用后状态更新"}

- [ ] 2.3 {任务描述}
  - 文件: {file.ts}
  - 依赖: 2.2
  - 完成标准: {如 "store action 可调用，loading 状态正常切换"}
  - 测试: {如 "加载中/成功/失败三种状态切换正确"}

### 阶段三：组件实现
- [ ] 3.1 {组件名}
  - 文件: {component.vue}
  - 依赖: 2.3
  - 完成标准: {如 "浏览器渲染正常，Props 按技术规格定义接收"}

- [ ] 3.2 {组件名}
  - 文件: {component.vue}
  - 依赖: 2.3
  - 完成标准: {...}

  - 文件: {file.vue}
  - 依赖: 全部组件
  - 完成标准: {如 "页面完整渲染，各区块数据正常显示"}

## 验证节点
| 节点 | 时机 | 验证方式 |
|------|------|---------|
| V1 | 阶段一完成后 | `pnpm vitest run tests/api/` 全绿 + curl 端点确认 |
| V2 | 阶段三每个组件完成后 | 浏览器查看组件渲染 |
| V3 | 阶段四完成后 | `pnpm vitest run` 全量通过 + /check-page + /perf-check |

---

## [决策] 决策点
- [决策] {需要人工判断的问题描述，如 "表单布局方案A还是方案B？"}

---

## 风险评估
- {可能的影响和应对}

## 测试计划
| 范围 | 测试类型 | 框架 | 覆盖目标 |
|------|---------|------|---------|
| 后端 API | 集成测试 | vitest + supertest | 每个端点：正常+校验+错误+边界 |
| Store/Service | 单元测试 | vitest | 业务逻辑：状态流转+计算+转换 |
| 编码前 | 类型检查 | tsc --noEmit | 类型与 schema 对齐 |
| 提交前 | 全量回归 | vitest run | 所有测试通过（不允许跳过） |
```

### 完成标准编写规则

每条完成标准必须是**可自动验证的具体条件**，禁止模糊描述：

| 错误（模糊） | 正确（可验证） |
|:------------|:--------------|
| "API 正常工作" | "curl /api/v1/xxx 返回 200，response body 包含 {field}" |
| "组件渲染正常" | "浏览器访问 /xxx，{element} 可见，无 console.error" |
| "类型对齐" | "npx tsc --noEmit 通过，schema-check 无报错" |
| "Store 可用" | "store.action() 调用后 state.xxx 正确更新" |

每个任务必须包含**测试字段**，描述该任务需要哪些测试保障：

| 任务类型 | 测试字段示例 |
|---------|------------|
| 后端 API | "POST 正常201 + 缺参数400 + 无权限403 + 重复409" |
| Service | "正常路径 + 空输入 + 边界值 + 异常抛出" |
| Store | "初始状态 + action后状态 + loading/error切换" |
| 组件 | "tsc 通过 + 渲染无报错（check-page 覆盖）" |

### 第3步：STOP

**不要写任何业务代码！** 将 PLAN.md 展示给用户，等待 `[Approved]`。

用户可以：
- 输入 `[Approved]` 开始自动施工
- 修改计划中的任何内容后输入 `[Approved]`
- 输入调整指令（如"把 3.1 和 3.2 合并"、"在阶段二加一个任务"）

---

## Phase 2: 自动施工

收到 `[Approved]` 后，严格按照施工图逐任务执行。

### 执行循环

```
FOR each 阶段 (一→一T→二→三→四):
  FOR each 任务（按依赖顺序）:
    1. 执行任务（创建/修改文件）
    2. 验证完成标准
       - IF 通过 → 继续步骤3
       - IF 失败 → 进入自动修复
    3. 运行测试（红绿循环）
       a. 如果任务有测试要求 → 先写测试（或已有测试）
       b. 运行测试: pnpm vitest run {相关测试文件}
       c. IF 全部通过(绿) → 标记 [x]，继续下一个
       d. IF 失败(红) → 自动修复循环:
          - 分析失败原因
          - 修复代码（不是修改测试来通过）
          - 重新运行测试
          - IF 通过 → 标记 [x]，继续
          - IF 第3次仍失败 → **暂停**，进入暂停处理
    4. 自动修复循环（最多3次，适用于完成标准验证失败）:
       - 分析失败原因
       - 修复代码
       - 重新验证完成标准 + 运行测试
       - IF 通过 → 标记 [x]，继续下一个
       - IF 第3次仍失败 → **暂停**，进入暂停处理
  阶段全部完成 → 执行本阶段验证节点 → 汇报阶段进度
```

### 测试执行规则

1. **禁止修改测试来让测试通过** — 只能修改实现代码
2. **每个任务完成后至少运行一次相关测试** — 即使任务没有"测试"字段，也要运行 `pnpm vitest run` 确认没有引入回归
3. **后端任务必须带 API 集成测试** — 覆盖：正常路径、参数校验、权限检查、错误处理
4. **Store/Service 任务必须带单元测试** — 覆盖：状态流转、计算逻辑、数据转换
5. **纯展示组件** — 用类型检查 + /check-page 替代测试

### 暂停处理（必须停下来等用户）

以下情况立即暂停，不继续下一个任务：

| 暂停条件 | 处理方式 |
|---------|---------|
| 自动修复3次仍失败 | 汇报失败原因 + 测试输出 + 已尝试的修复，等待用户指示 |
| 测试3次仍红 | 汇报失败的测试用例 + 错误信息，等待用户指示 |
| 遇到 `[决策]` 标记 | 呈现问题 + 选项，等待用户选择 |
| Tech Spec 有矛盾或遗漏 | 列出矛盾点，建议修正方案，等待确认 |
| 完成标准无法验证 | 说明环境问题或标准不清晰，等待指示 |
| 发现影响已有功能的风险 | 列出风险点，等待确认是否继续 |
| 测试引入回归（已有测试变红） | 定位回归原因，修复后重新运行全量测试 |

### 汇报格式

**阶段完成汇报**：
```
阶段 X 完成（N/N 任务通过）
  ✅ 1.1 {任务名}
  ✅ 1.2 {任务名}
验证节点 VX: 通过/失败
```

**暂停汇报**：
```
⏸ 暂停于 任务 X.Y: {任务名}
原因: {具体原因}
已尝试: {3次修复的简要描述}
建议: {下一步建议}
```

**全部完成汇报**：
```
施工完成（N/N 任务通过，耗时约 Xm）
验证: /check-page → 执行中...
```

### 提交策略

- 每个阶段完成后自动 `git commit`
- **提交前必须 `pnpm vitest run` 全量通过** — 不允许提交有测试失败或回归的代码
- 提交消息格式: `feat({模块}): {阶段描述}`
- 如果某个任务独立且不影响其他任务，也可以单独提交

## TDD 强制范围（高风险业务逻辑先写测试再写实现）

以下代码必须先写测试、再写实现，禁止先实现后补测试：
- **调度管道**: 去重、拆分、拼单、衔接、派单（5个服务）
- **计价/报价逻辑**: 价格计算、费用拆分、结算
- **编号生成服务**: 唯一性、格式、校验位
- **任务状态流转**: 状态机转换、权限校验
- **结算计算**: 分账、对账、手续费

UI 模板和纯展示组件不要求 TDD，用 `/check-page` 对照设计稿验证即可。

## 技术选型表（自动使用，不询问用户）

| 场景 | 选择 |
|:-----|:-----|
| 组件库 | Ant Design Vue |
| 状态管理 | Pinia |
| 日期处理 | dayjs |
| 表单验证 | Ant Design Form |
| API调用 | REST + async/await |
| 中文搜索 | `filterOption=false` + `@search` + 拼音首字母 |
| 大数据表格 | `:virtual="true"` 虚拟滚动 |
| 分页组件 | 放在表格 overflow 容器外部（防止裁切） |

## 与其他技能的关系

| 技能 | 关系 |
|:-----|:-----|
| `/tech-spec` | 上游，本技能读取其产出生成施工图 |
| `/wf-new` | 编排器，调用本技能完成编码阶段 |
| `/plan` | 并行存在，`/plan` 服务于 `/wf-batch` 的批量多页面任务 |
| `/check-page` | 阶段四的验证手段 |
| `/wf-batch` | 不参与本流程，仅用于独立的多页面批量开发 |
