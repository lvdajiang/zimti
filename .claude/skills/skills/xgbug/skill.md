# 系统化修bug工作流 `/xgbug`

> **核心原则**：修一个bug，消灭一类bug。禁止"打补丁"式修复。

当用户报告bug时，**必须严格按以下6步顺序执行**。每一步完成后向用户汇报，等待确认后再进入下一步。

---

## 步骤1：确认问题（强制）

**目标**：确保修对东西，避免返工。

执行内容：
1. **复述问题**：向用户确认具体是哪个页面/组件/接口
2. **确认症状**：描述具体的错误表现（报错信息、截图、URL）
3. **确认边界**：这个bug在什么条件下出现？什么时候不出现？
4. **等待用户确认**

**禁止**：跳过确认直接开始修复。

**输出格式**：
```
🔍 问题确认
- 页面/组件：[具体位置]
- 错误表现：[具体症状]
- 复现条件：[触发条件]
请确认以上理解是否正确？
```

---

## 步骤2：找根因（深入诊断）

**目标**：找到问题的根本原因，而非表面现象。

执行内容：
1. 阅读报错信息/控制台输出，定位出错的代码行
2. 沿调用链向上追溯，找到触发错误的源头
3. 区分三类根因：
   - **配置问题**（CORS、端口、环境变量）
   - **代码逻辑问题**（空值、类型错误、边界条件）
   - **数据问题**（格式不一致、缺失、脏数据）
4. 用一句话总结根因

**禁止**：只看报错行就动手改。报错行往往是受害者，不是加害者。

**输出格式**：
```
🔬 根因分析
- 报错位置：[文件:行号]
- 根本原因：[一句话描述]
- 根因类型：配置问题 / 代码逻辑 / 数据问题
- 证据：[为什么确定是这个原因]
```

---

## 步骤3：扫描同类问题（关键步骤）

**目标**：找出项目中所有存在相同根因的地方，一次性全部修复。

执行内容：
1. 根据根因类型，搜索项目中同类代码：
   - **配置问题** → 搜索所有配置文件、环境变量引用
   - **代码模式问题** → `grep` 搜索相同的代码模式
   - **组件使用问题** → 搜索所有使用同类组件的地方
   - **API/排序问题** → 搜索所有类似的API端点或排序逻辑
2. 列出所有需要修改的文件和位置
3. 评估每处修改的风险和影响范围

**禁止**：只修当前出问题的文件就收工。

**输出格式**：
```
🔎 同类问题扫描
发现 N 处同类问题：

| # | 文件 | 行号 | 问题 | 风险 |
|---|------|------|------|------|
| 1 | xxx | L42 | xxx | 低 |
| 2 | xxx | L108 | xxx | 中 |

本次将全部修复。
```

---

## 步骤4：修复所有问题

**目标**：一次性修复步骤3中列出的所有问题。

执行内容：
1. **优先选择统一方案**：如果有封装/抽象的可能，优先封装（如创建公共组件、统一工具函数）
2. 按文件逐个修复，每修复一处说明改了什么、为什么
3. 修复后检查是否引入新问题（如：改了这个组件是否影响其他使用它的页面）
4. 每个修复文件单独 git commit，commit message 格式：`fix: [一句话描述根因和修复]`

**禁止**：
- 修A的时候顺手改B（不相关的改动）
- 不说明改了什么就提交
- 一次性提交所有文件的修改（无法定位回滚）

**输出格式**：
```
🔧 修复完成
已修复 N 处问题：

- [文件1]: 改了什么，为什么
- [文件2]: 改了什么，为什么

新增公共组件/工具：[如有]
```

---

## 步骤5：写回归测试（防止复发）

**目标**：确保这个bug不会再出现，也确保修复没有破坏其他功能。

执行内容：
1. **为本次bug写一个复现测试**：
   - 模拟触发bug的条件
   - 断言修复后不再出现该问题
2. **检查相关功能的现有测试**是否仍然通过：
   - 运行 `pytest` 或相关测试命令
   - 如有失败，立即修复
3. **如果项目无测试框架**，至少做以下手动验证：
   - 在出问题的页面完整操作一遍核心流程
   - 在相关页面检查是否有同类问题
   - 在浏览器控制台检查是否有新的报错

**禁止**：声称修复完成但不做任何验证。

**输出格式**：
```
✅ 验证结果
- 回归测试：[通过/失败，附测试结果]
- 核心流程验证：[列出验证的操作和结果]
- 浏览器控制台：[无新报错 / 列出新报错]
- 边缘案例：[列出已验证的边界条件]
```

---

## 步骤6：部署

**目标**：将修复推送到生产环境。

执行内容：
1. 确认所有测试通过
2. 执行部署（调用 bushu 技能或手动部署）
3. 部署后在生产环境快速验证核心功能
4. 如有问题，立即回滚

**输出格式**：
```
🚀 部署完成
- 部署时间：[时间]
- 生产验证：[结果]
- 回归风险：[低/中/高]
```

---

## 违规纠正

如果 Claude 在执行过程中跳过了某个步骤，用户应指出并要求回到该步骤。

如果用户说"快速修复"或"紧急"，**仍然必须执行步骤1（确认）和步骤3（扫描同类）**，可以简化步骤5（测试）。
