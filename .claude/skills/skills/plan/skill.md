# 工作计划 `/plan`

> **使用场景**：手动编写或用AI生成 wf-batch 可识别的任务计划。
> **目标**：花时间制定计划，交给 /wf-batch 自动执行。

## 用法

```
/plan                           # 交互式：引导你逐步创建计划
/plan 审查client, 修复排序bug   # 快速生成：一句话转计划
/plan 文件                      # 从文件读取计划模板
```

---

## 计划格式标准

### 行格式

每行一个任务，用方括号标记类型：

```
[类型] 任务描述
```

### 支持的类型标记

| 标记 | 执行策略 | 示例 |
|------|---------|------|
| `[审查]` | `/code-review` 扫描代码质量 | `[审查] client前端代码` |
| `[修复]` | `/xgbug` 系统化修bug | `[修复] 排序bug` |
| `[开发]` | `/wf-dev` 开发新页面 | `[开发] 订单管理页面` |
| `[审查+修复]` | `/wf-review` 审查并修复 | `[审查+修复] admin前端` |
| `[算法]` | `/wf-algo` 修改管道算法 | `[算法] dispatch服务` |
| `[部署]` | `/wf-deploy` 部署到生产 | `[部署]` |
| `[检查]` | `/deploy-check` 部署前检查 | `[检查]` |
| `[性能]` | `/perf-check` 性能巡检 | `[性能] client列表页` |
| `[组件]` | `/component-check` 组件一致性 | `[组件] 表格组件` |
| `[页面]` | `/check-page` 对照设计稿检查 | `[页面] 派单管理` |

### 完整计划示例

```markdown
## 工作计划 2026-04-19

[审查] client前端代码质量
[修复] combination_service.py 排序bug
[修复] connection_service.py 排序bug
[审查+修复] admin前端代码
[检查]
```

### 分批计划示例

```markdown
## 工作计划 2026-04-19

### 白天批（不部署）
[审查] client前端代码质量
[修复] combination_service.py 排序bug
[修复] connection_service.py 排序bug
[审查+修复] admin前端代码
[检查]

### 晚上批（部署）
[部署]
```

### 高级用法：指定范围

在任务描述中包含文件路径或URL，`/wf-batch` 会自动定位：

```
[审查] frontend/src/views/client/          → 指定目录
[修复] backend/app/services/dispatch.py    → 指定文件
[开发] /admin/order-management              → 指定页面URL
[算法] dedup                                → 指定管道服务
```

### 高级用法：指定模型

用 `@模型名` 强制指定某个任务使用的模型：

```
[审查] client前端代码质量                    → 自动推荐 glm-4.7
[修复] dispatch_service.py 排序bug @glm-5.1  → 强制用 glm-5.1
[部署] @glm-5.1                             → 部署用强模型
```

## 执行步骤

### 模式1：快速生成（有参数时）

用户输入一句话，自动转为计划格式：

```
你：/plan 审查client, 修复排序bug, 部署

我：
📋 工作计划

1. [审查] client前端代码
2. [修复] 排序bug
3. [部署]

确认后可直接执行：/wf-batch [粘贴上面的计划]
```

### 模式2：交互式创建（无参数时）

逐步引导用户创建计划：

**第1步：选择场景**

```
你想做什么？
1. 代码质量巡检（审查+修复）
2. bug修复
3. 页面开发
4. 算法优化
5. 混合任务（自定义）
6. 从待处理清单生成
```

**第2步：选择范围**

```
影响范围？
1. client 前端
2. admin 管理后台
3. agency 旅行社端
4. backend 后端
5. 全部
```

**第3步：生成计划**

根据选择，读取相关文件/目录信息，生成结构化计划：

```
📋 工作计划（共N项）

1. [审查] client前端代码质量
2. [审查] admin前端代码质量
3. [修复] 后端排序bug（2处）
4. [检查]
5. [部署]

复制以上计划，随时执行：/wf-batch
```

### 模式3：从文件读取

```
你：/plan docs/my-plan.md

我：读取文件并解析为任务列表
```

### 模式4：从待处理清单生成

```
你：/plan auto

我：读取 session_progress.md 和 known_issues.md，自动生成计划
```

## 与 /wf-batch 的关系

```
/plan → 生成计划
  ↓
/wf-batch [粘贴计划] → 执行计划
```

`/wf-batch` 接受以下输入格式：
1. `/wf-batch 审查client, 修复bug` — 逗号分隔的简写（自动加标记）
2. `/wf-batch` + 粘贴计划文本 — 完整计划
3. `/wf-batch auto` — 从待处理清单读取

### 简写自动映射

`/wf-batch` 也支持不带标记的简写，自动映射：

| 简写关键词 | 自动映射为 |
|-----------|-----------|
| 审查/review | `[审查]` |
| 修复/fix/bug | `[修复]` |
| 开发/dev/页面 | `[开发]` |
| 算法/algo | `[算法]` |
| 部署/deploy/上线 | `[部署]` |
| 检查/check | `[检查]` |

所以以下两种写法等价：

```
/wf-batch [审查]client, [修复]排序bug, [部署]
/wf-batch 审查client, 修复排序bug, 部署
```

---

## 注意事项

- 计划中的任务按顺序执行，前面的任务结果会影响后面的
- 建议把 `[检查]` 放在 `[部署]` 之前
- `[修复]` 任务如果需要定位具体文件，最好在描述中包含文件名
- 计划可以保存为文件，方便复用和版本管理
