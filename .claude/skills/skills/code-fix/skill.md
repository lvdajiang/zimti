# 代码修复 `/code-fix`

> **使用场景**：已知问题清单，需要逐项修复。典型搭配：`/code-review` 发现问题后调用。
> **目标**：按清单逐项定位代码、修改、验证，确保不遗漏、不引入新问题。

## 用法

```
/code-fix                           # 交互式：从上次 code-review 报告中读取问题清单
/code-fix client                    # 指定范围，先自动扫描再修复
/code-fix 1,2,5                     # 只修复清单中第1、2、5项
/code-fix --severity=high           # 只修复严重级别的问题
```

---

## 执行步骤

### 第1步：加载问题清单（5秒）

**目标**：明确要修什么。

执行内容：
1. **有参数时**：根据参数确定范围，自动执行 `/code-review` 的6类扫描，生成问题清单
2. **无参数时**：从会话上下文中找到最近的 `code-review` 报告，提取问题清单
3. **有编号参数时**（如 `1,2,5`）：从清单中筛选指定项
4. **有 `--severity` 参数时**：按严重度筛选（`high`=❌高, `medium`=⚠️中, `low`=💡低）

确认后展示待修复清单，等待用户确认：

```
📋 待修复清单（共N项）

| # | 严重度 | 类别 | 文件:行号 | 问题 | 修复方案 |
|---|--------|------|----------|------|---------|
| 1 | ❌高   | 性能 | xxx:L42  | pageSize=1000 | 改为后端分页 |
| 2 | ⚠️中   | 风格 | xxx:L15  | fetch而非api | 改用@/api/xxx |

确认修复以上N项？（可指定序号筛选）
```

**用户确认后进入第2步。**

---

### 第2步：逐项修复（核心步骤）

**目标**：每个问题定位代码 → 修改 → 验证。

对清单中的每个问题，执行以下3个子步骤：

#### 2.1 定位代码

1. 读取目标文件，定位到问题行号
2. 理解上下文（前后10行代码）
3. 确认问题确实存在（排除误报）
4. 如果是误报，标记跳过并说明原因

```
📌 定位问题 #N — [问题描述]
- 文件：[文件路径]
- 行号：L[行号]
- 上下文：[当前代码的关键片段]
- 状态：确认存在 ✅ / 误报跳过 ⏭️（原因：xxx）
```

#### 2.2 执行修改

1. 根据修复方案修改代码
2. 遵循项目技术规范（见 CLAUDE.md）：
   - 统一使用 `@/api/xxx` 封装
   - 错误处理用 `message.error`
   - 编号通过后端服务生成
   - 禁止 `console.log` 残留
3. 如果发现同类问题（同一文件或关联文件），一并修复并记录

```
🔧 修复问题 #N
- 改了什么：[具体改动]
- 为什么这样改：[理由]
- 同类问题：一并修复了 [其他位置] / 无
```

#### 2.3 快速验证

修改后立即做最小验证：
1. 确认修改没有语法错误（文件可正常解析）
2. 确认没有引入新的依赖问题（import 是否正确）
3. 确认修改没有破坏关联逻辑（相关函数是否仍能正常工作）

```
✅ 验证通过 #N
- 语法检查：通过
- 依赖检查：通过
- 关联检查：通过
```

**每修复一项，更新进度**：
```
进度：[已修复N/共M项] ✅
```

---

### 第3步：修复总结（10秒）

**目标**：汇报修复结果。

执行内容：
1. 统计修复结果
2. 列出跳过的误报项（如有）
3. 列出修复涉及的文件清单
4. 建议下一步操作

```
📋 修复报告

### 结果统计
- 总计：N项
- 已修复：N项
- 跳过（误报）：N项
- 修复失败：N项（如有）

### 修复文件清单
- [文件1]: 修复了N个问题
- [文件2]: 修复了N个问题

### 跳过的误报
- #N: [原因]

### 建议下一步
- /wf-deploy  → 部署修复
- /code-review → 再次审查确认
```

## 问题类型修复指南

不同类型的问题有标准修复方式：

| 问题类型 | 标准修复方式 | 注意事项 |
|---------|------------|---------|
| **pageSize超标** | 改为分页参数，加 `pagination` | 检查是否需要后端配合 |
| **catch无提示** | 加 `message.error(e.message)` | 区分网络错误和业务错误 |
| **fetch混用** | 改为 `@/api/xxx` 封装 | 检查是否已有封装方法 |
| **console.log** | 直接删除 | 确认不是必要的调试日志 |
| **mode="tags"** | 改为 `filterOption=false` + `@search` | 参考中文输入法搜索方案 |
| **localStorage** | 改为后端API | 评估是否需要迁移数据 |
| **死代码** | 删除未使用的函数/导入 | 确认确实是未使用 |
| **空函数体** | 实现逻辑或删除 | 不能留空 |
| **假操作** | 加真实的API调用 | 不能只提示成功 |

## 与其他 skill 的关系

| Skill | 用途 | 区别 |
|-------|------|------|
| `/code-fix` | 按已知清单逐项修复 | 问题已知，直接改 |
| `/xgbug` | 诊断并修复单个功能bug | 需要先确认问题和找根因 |
| `/code-review` | 扫描发现问题 | 只发现问题不修复 |
| `/simplify` | 审查改动代码的质量 | 审查单次改动，不是批量修复 |

## 典型工作流

```
/code-review → 发现问题清单 → /code-fix 1,2,3 → 修复指定项 → /wf-deploy
/code-review → 发现问题清单 → /code-fix --severity=high → 修复严重项 → /wf-deploy
```

## 注意事项

- 修复前必须确认问题确实存在，避免误报导致错误修改
- 每修完一项立即验证，不要攒到最后一起检查
- 发现同类问题时一并修复，但要向用户说明
- 如果修复某项时发现新问题，追加到清单末尾，不中断当前流程
- 超过3个文件时按文件分组提交，方便回滚
