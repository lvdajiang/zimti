# 对齐理解 `/align`

> **使用场景**：开始任何非平凡任务前，或讨论中发现理解可能不一致时。
> **前置输入**：如果已有技术规格（`05-技术规格/`）或需求文档（`01-需求/`），先读取作为对齐依据。
> **目标**：用7个关键问题确保双方对问题的理解完全一致，防止"修错了东西"。

## 用法

```
/align                           # 对齐当前讨论的问题
/align 资源库保存后数据没显示      # 带问题描述，直接进入对齐
```

## 什么时候用

**必须用（自动触发）**：
- 用户报告bug或问题时（在 `/xgbug` 步骤1之前）
- 用户提出新功能需求时
- 任务涉及3个以上文件时
- 讨论中出现"你是不是想说..."的迹象时

**建议用**：
- 用户说"XX有问题"但描述模糊时
- 需求可能有多种理解方式时
- 修改前需要确认影响范围时

**不需要用**：
- 任务明确且单一（如"把标题改成XX"）
- 用户已经给出了详细且唯一的描述
- 纯技术实现细节的选择

## 7个关键问题

收到用户的问题或需求后，**依次确认以下7个问题**：

### 1. 是什么？— 明确问题本身

确认用户描述的现象，用自己的话复述。

- ❌ "我知道了" → 直接开始改
- ✅ "你说的'资源库有问题'，具体是：保存后实体类型和子类列没有显示数据，对吗？"

### 2. 在哪里？— 定位范围

确认问题发生的具体位置。

- 页面/组件名称
- 前端 or 后端 or 数据库
- 具体URL（如果适用）

### 3. 什么条件下出现？— 触发条件

确认问题何时发生，何时不会发生。

- 操作步骤
- 数据条件（特定类型/状态的数据才触发？）
- 环境条件（本地/测试/生产？）

### 4. 应该是什么样？— 预期结果

确认修复后的正确行为。

- 正确的显示/行为是什么
- 有没有参考页面（"XX页面是正确的"）

### 5. 影响范围多大？— 评估严重度

确认问题的边界。

- 只影响某个具体页面，还是全局问题
- 只影响特定数据，还是所有数据
- 是否阻塞其他功能

### 6. 关联的其他问题？— 防止漏修

确认是否有相关的其他问题。

- 同类组件/页面是否也有
- 新建 vs 编辑 是否都有
- 相关的上下游功能是否受影响

### 7. 已经试过什么？— 避免重复劳动

确认用户是否已经尝试过解决。

- 已经做了什么操作
- 查过的资料或参考
- 之前修过类似的吗

## 执行方式

### 方式一：一次性确认（问题明确时）

如果用户已经给出了大部分信息，一次性列出确认清单：

```
📋 问题对齐

1. 是什么：保存后实体类型和子类列没有显示
2. 在哪里：agency 端资源库编辑页面（http://localhost:3001/agency/resource）
3. 什么条件下：编辑已有资源并保存时（新建不受影响）
4. 应该是什么样：保存后显示实体类型和子类
5. 影响范围：所有实体类型都受影响
6. 关联问题：未知，需要检查
7. 已尝试：刷新页面，没用

以上理解正确吗？有需要补充的吗？
```

### 方式二：逐项确认（问题模糊时）

如果用户描述模糊，逐步追问：

```
我想先确认几个问题，确保理解正确：

1. 你说的"资源库有问题"，具体是什么表现？是数据显示不出来，还是保存失败？
```

每确认一项再问下一项，避免一次问太多让用户烦躁。

## 判断规则

**核心原则：任务明确时自动跳过，不问废话。**

| 用户输入 | 策略 |
|---------|------|
| 详细且唯一（如"第42行缺少分号"） | 跳过对齐，直接修复 |
| 给了具体文件路径（如"fix ResourceTable.vue"） | 跳过对齐，直接修复 |
| 给了具体URL（如"http://39.97.59.179/admin/xxx"） | 跳过对齐，直接处理 |
| 给了具体错误信息（如"TypeError: xxx"） | 跳过对齐，直接诊断 |
| 有一定细节但可能多解 | 快速确认1-2个关键点 |
| 模糊（如"XX有问题""看看XX"） | 完整7问对齐 |
| 涉及业务逻辑变更 | 必须完整对齐 |
| 用户明确说"快速做""不用问我" | 跳过对齐，用最佳判断直接执行 |

**自动跳过清单**（以下情况不需要对齐）：
- 用户输入包含文件路径
- 用户输入包含URL
- 用户输入包含错误信息/堆栈
- 用户输入包含具体的组件/页面名称
- 调用 `/wf` 系列工作流时（工作流内部自行处理对齐）
- 调用 `/code-fix` 时（问题清单已明确）

## 第8步：需求穷举清单（复杂任务必做）

复杂任务（涉及业务逻辑、多角色交互、状态流转）在7问对齐后，必须枚举以下场景：

### 正常流
- 主路径的完整执行流程，从触发到完成

### 异常流
- 数据缺失（必填字段为空、关联数据不存在）
- 权限不足（无操作权限、跨机构访问）
- 网络异常（API超时、服务不可用）
- 并发冲突（同一资源同时被多个请求操作）

### 边界值
- 空数据（列表为空、查询无结果）
- 超大数据（列表超100条需虚拟滚动、文本超长截断）
- 临界数值（金额为0、数量为1、日期跨月/跨年）
- 时间边界（23:59→00:00、月初月末、节假日）

### 并发场景
- 同一资源同时被多个请求操作（乐观锁/悲观锁）
- 同一用户重复提交（防抖/幂等）

枚举完成后，将覆盖的场景标注到计划中，确保实现时不遗漏。

## 对齐后的输出

确认完成后，输出一句话总结：

```
✅ 对齐完成：[一句话总结问题和预期]
```

然后进入下一个步骤（修复/规划/开发等）。
