# 需求细化 `/refine`

> **使用场景**：有一个模糊的想法或功能方向，但细节不明确，需要系统化地细化成具体需求。
> **目标**：把"我觉得需要XX"变成"XX页面有12个字段、3个按钮、这些数据从哪来"。
> **位置**：在 `/design-spec` 之前执行，是整个工具链的第一步。

## 用法

```
/refine 我想做一个司机评价系统
/refine 旅行社需要一个报价管理功能
/refine 客户端要能查看历史订单
```

## 核心原则

**写不出来 = 没想清楚。** 遇到写不下去的地方，停下来思考，不要跳过。
**不写代码。** 这个阶段只产出需求文档，不涉及技术实现。

---

## 4个步骤（必须按顺序执行）

### 第1步：一句话定义（5分钟）

把想法写成一句话，格式固定：

```
[谁] 需要 [什么功能] 来解决 [什么问题]
```

检验标准：拿掉任何一个部分，句子就不完整。
如果写不出来，说明想法还不够成型，**不要进入下一步**。

输出示例：
```
旅行社 需要一个报价管理功能 来解决手工算价容易出错且效率低的问题
```

### 第2步：用户故事列表（30分钟）

列出所有可能的用户场景。**不考虑技术实现，只写用户能做什么。**

格式：
```
作为 [角色]，我想要 [做某事]，以便 [达到某目的]
```

分类列出：
- **核心故事**（必须有，否则功能不成立）
- **重要故事**（显著提升体验）
- **锦上添花**（有更好，没有也能用）

检验标准：核心故事 < 5个。如果超过5个，说明功能范围太大，需要拆分。

输出示例：
```
核心故事：
- 作为旅行社，我想要选择产品和日期自动计算价格，以便快速给客户报价
- 作为旅行社，我想要保存报价单并发送给客户，以便客户确认

重要故事：
- 作为旅行社，我想要对比不同产品的报价，以便推荐性价比最高的方案
- 作为管理员，我想要查看所有旅行社的报价统计，以便分析业务趋势

锦上添花：
- 作为旅行社，我想要一键生成报价PDF，以便打印给客户
```

### 第3步：页面清单 + 逐页功能说明（1-2小时）

每个用户故事对应一个或几个页面。先列出页面清单，再为每个页面写功能说明。

#### 3.1 页面清单

```
| 页面名称 | 对应用户故事 | 页面类型 |
|---------|-------------|---------|
| 报价列表 | 查看报价统计 | 列表页 |
| 新建报价 | 创建报价单 | 表单页 |
| 报价详情 | 查看/发送报价 | 详情页 |
```

#### 3.2 逐页功能说明（核心产出）

对每个页面，按以下模板描述功能。**这是整个功能说明文档的核心**，后续 `/design-spec`（UI规格）、`/data-flow`（数据流）、`/tech-spec`（技术规格）都以此为输入。

```
### 页面：{页面名称}

**功能概述**：一句话描述这个页面让用户做什么

**页面分区**：

| 区块名称 | 功能说明 | 包含元素 |
|---------|---------|---------|
| 顶部导航 | 返回上一页，显示页面标题 | 返回按钮、标题文字 |
| 筛选标签栏 | 按状态筛选列表 | 标签：全部/待报价/已发送/已过期 |
| 报价列表 | 展示报价卡片，支持下拉刷新和上拉加载 | 卡片列表，每张卡片显示摘要信息 |
| 悬浮按钮 | 快速创建新报价 | + 按钮 |

**显示字段**：

| 字段 | 显示方式 | 数据来源 | 备注 |
|------|---------|---------|------|
| 报价编号 | 文字，只读 | 后端生成 | 如 QT-20260308-001 |
| 客户名称 | 文字，只读 | 用户选择 | — |
| 报价金额 | 文字，只读，加粗 | 后端计算 | ¥ 格式 |
| 状态 | 标签（颜色区分） | 后端存储 | 待报价/已发送/已接受/已过期 |
| 创建时间 | 文字，只读 | 后端存储 | yyyy-MM-dd |

**操作定义**：

| 操作 | 触发方式 | 行为 | 权限/条件 |
|------|---------|------|----------|
| 查看详情 | 点击卡片 | 跳转报价详情页 | 所有 |
| 新建报价 | 点击悬浮按钮 | 跳转新建报价页 | 有创建权限 |
| 删除报价 | 左滑卡片 | 弹出确认框→删除 | 仅草稿状态 |
| 下拉刷新 | 手势下拉 | 重新加载列表 | 所有 |
| 上拉加载 | 滚动到底部 | 加载下一页 | 还有更多数据时 |

**页面状态**：

| 状态 | 表现 |
|------|------|
| 加载中 | 显示骨架屏 |
| 空数据 | 显示提示文字 + 引导按钮 |
| 加载失败 | 显示错误提示 + 重试按钮 |

**业务规则**：

- 默认按创建时间倒序排列
- 过期报价自动标记灰色
- 草稿超过30天提示是否删除
```

#### 3.3 页面导航关系

用流程图描述页面间的跳转关系：

```
报价列表 ──点击卡片──→ 报价详情
   │                      │
   └──点击+按钮──→ 新建报价 ──保存成功──→ 报价详情
```

#### 3.4 数据关系草图

列出这个功能涉及的核心实体及其关系：

```
产品 1──N 报价单 1──N 报价明细
                 │
                 └──N 客户
```

检验标准：
1. 拿着每个页面的功能说明，一个不了解项目的人能说出这个页面**做什么**吗？
2. 每个显示字段都有来源说明吗？
3. 每个操作都有触发方式和行为描述吗？

### 第4步：边缘场景补充

对每个页面的核心操作，补充边缘场景：

```
页面：新建报价

### 正常流
选择产品 → 选日期 → 填人数 → 计算报价 → 发送

### 异常流
- 产品在该日期无库存 → 提示"该日期已满"
- 人数超过车辆容量 → 提示"需要拆分为多辆车"
- 网络断开 → 本地暂存，恢复后提示是否继续

### 边界值
- 人数为1 → 正常（包车）
- 日期选择过去日期 → 禁止选择
- 价格为0 → 提示确认"该产品免费？"
```

## 输出

完成4步后，输出一份结构化的**功能说明文档**（包含需求 + 逐页功能说明）：

```
📋 功能说明文档 — [功能名称]

### 一句话定义
[第1步产物]

### 用户故事
[第2步产物]

### 页面清单
[第3步产物 — 页面清单表格]

### 逐页功能说明（核心产出）
[第3步产物 — 每个页面的完整功能说明]
  - 功能概述
  - 页面分区（区块名称 + 功能说明 + 包含元素）
  - 显示字段（字段 + 显示方式 + 数据来源 + 备注）
  - 操作定义（操作 + 触发方式 + 行为 + 权限/条件）
  - 页面状态（加载中 / 空数据 / 加载失败）
  - 业务规则

### 页面导航关系
[第3步产物 — 页面跳转流程图]

### 数据关系草图
[第3步产物 — 核心实体关系]

### 边缘场景
[第4步产物]

### 待确认问题
- [列出第3步中标注"需要确认"的项]

```

然后**保存到文件**：`开发文档/01-需求/{功能名}-需求.md`

输出确认：
```

📊 覆盖：一句话定义 / 用户故事 / 逐页功能说明 / 边缘场景
📋 每个页面包含：功能概述 / 页面分区 / 显示字段 / 操作定义 / 页面状态 / 业务规则
→ 下一步：/design-spec [页面名称]（基于功能说明生成7要素设计规格）
```

## 与其他技能的关系

```
/refine → /design-spec → /design-detail → /design-page → /data-flow → /tech-spec → /align → /plan-first → 编码
 想法→需求   需求→设计规格   设计规格→页面描述  页面描述→.pen稿  数据流     数据流→技术蓝图   确认边界   制定计划
```

- `/refine` 产出功能说明文档（保存到 `01-需求/`），`/design-spec` 读取功能说明生成7要素设计规格
- `/refine` 的第3步（逐页功能说明）是 `/design-spec` 确定页面类型、分区、字段的依据
- `/refine` 的第3步（显示字段表）是 `/data-flow` 字段溯源的输入
- `/refine` 的第4步（边缘场景）是 `/align` 需求穷举清单的输入

## 执行原则

1. **每步必须产出文字**，不允许只在脑子里想
2. **遇到不确定的标记为"待确认"**，不要猜测
3. **范围宁窄勿宽**，先做核心故事，锦上添花的后续迭代
4. **如果第2步核心故事超过5个**，提醒用户拆分成多个功能分别细化
