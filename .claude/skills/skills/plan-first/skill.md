# 规划-执行分离模式

## Phase 1: 调研与计划

**前置读取**（如果存在）：
- `开发文档/05-技术规格/{页面名}-技术规格.md` → 组件层级、Props、API、表结构
- `开发文档/04-数据流/{页面名}-数据流.md` → API契约、字段溯源
- PLAN.md 应引用技术规格中的具体组件名、API函数名、数据库表名

1. 阅读相关代码和文档，理解当前上下文。
2. 创建 `PLAN.md` 文件，包含以下内容：
   - **目标**：本次要完成的功能/修复的问题
   - **涉及文件**：需要创建/修改的文件列表
   - **详细步骤**：按顺序列出具体操作（如"修改 model.py 添加字段"）
   - **风险评估**：可能的影响（如数据库迁移、API 变更）
   - **测试计划**：如何验证结果
3. **STOP**：不要写任何业务代码！等待用户输入 `[Approved]` 指令。

## Phase 2: 执行
收到 `[Approved]` 后，严格按照计划逐步实施，每完成一步提交一次 Git。

## TDD 强制范围（高风险业务逻辑先写测试再写实现）

以下代码必须先写测试、再写实现，禁止先实现后补测试：
- **调度管道**: 去重、拆分、拼单、衔接、派单（5个服务）
- **计价/报价逻辑**: 价格计算、费用拆分、结算
- **编号生成服务**: 唯一性、格式、校验位
- **任务状态流转**: 状态机转换、权限校验
- **结算计算**: 分账、对账、手续费

UI 模板和纯展示组件不要求 TDD，用 `/check-page` 对照设计稿验证即可。
