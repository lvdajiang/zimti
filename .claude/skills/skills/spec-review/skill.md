# 设计规格审查 `/spec-review`

> **使用场景**：`*-设计规格.md` 写完或修改后，检查7要素完整性和文档内部一致性。
> **目标**：确保设计规格文档覆盖所有7要素，且与需求文档和设计系统规范无冲突。
> **前置**：`/design-spec` 生成的设计规格文件

## 执行方式（Subagent 模式）

本技能通过 Subagent 执行分析，不占用主对话上下文。

**必须使用 Agent tool** 启动分析代理，传入以下参数：
- subagent_type: "general-purpose"
- prompt: 包含下方完整审查清单

代理在独立上下文中读取设计规格文件和基准文件，**只返回问题清单摘要给主对话**。禁止在主窗口中逐个读取设计规格文件。

## 用法

```
/spec-review                  # 审查所有设计规格
/spec-review 选题工作台        # 审查单个设计规格
```

## 执行步骤

### 第1步：收集文件（10秒）

扫描 `开发文档/02-设计/*-设计规格.md`，确定审查范围。
同时读取：
- `开发文档/01-需求/自媒体内容生产系统-需求.md`（需求基准）
- `开发文档/03-视觉/设计系统规范.md`（设计系统基准）

### 第2步：五维度审查（60秒）

#### 2.1 七要素完整性

设计规格必须包含以下7个章节，缺一不可：

| # | 要素 | 对应章节 | 检查内容 |
|---|------|---------|---------|
| 1 | 布局结构 | 第1章 | 整体布局图、尺寸规格表、布局规则 |
| 2 | 颜色映射 | 第2章 | 背景、文字、按钮、标签颜色全部引用变量 |
| 3 | 字体规范 | 第3章 | 字体家族、字号层级、字重应用、行高 |
| 4 | 间距规范 | 第4章 | 页面级、组件级间距全部用变量 |
| 5 | 组件规格 | 第5章 | 每个组件的属性表格 |
| 6 | 状态设计 | 第6章 | 组件状态、加载/空/错误状态 |
| 7 | 响应式规则 | 第7章 | 4个断点的适配方案 |

**检查规则**：

| 检查项 | 说明 |
|--------|------|
| 章节存在 | 7个章节是否都存在 |
| 子章节完整 | 每个章节的子项是否齐全 |
| 表格完整 | 每个属性表格是否有完整的列 |

#### 2.2 需求覆盖完整性

读取需求文档中对应页面的「显示字段」和「操作定义」，逐项核对设计规格的「附录：字段全覆盖清单」。

**检查规则**：

| 检查项 | 说明 |
|--------|------|
| 显示字段全覆盖 | 需求定义的每个字段在设计规格中有对应条目 |
| 操作全覆盖 | 需求定义的每个操作在设计规格中有对应条目 |
| 字段有样式 | 每个显示字段有字号、字重、颜色变量引用 |
| 操作有交互 | 每个操作有触发位置、交互方式、条件 |

#### 2.3 内部一致性

检查设计规格文档内部的矛盾：

| 检查项 | 说明 |
|--------|------|
| 尺寸不自洽 | 布局图中区域高度之和与总高度矛盾 |
| 颜色变量自洽 | 同一元素在不同章节引用的颜色变量一致 |
| 间距自洽 | 组件规格中的间距值与间距规范章节一致 |
| 状态完整 | 状态设计章节覆盖了组件规格中定义的所有状态 |
| 响应式覆盖 | 每个断点都声明了布局、宽度、间距 |

#### 2.4 设计系统合规性

检查设计规格引用的变量是否在 `设计系统规范.md` 中有定义。

**检查规则**：

| 检查项 | 说明 |
|--------|------|
| 变量存在 | `$brand-indigo` 等变量在设计系统中有定义 |
| 值一致 | 变量引用的值与设计系统定义的值一致 |
| 无硬编码 | 不应出现直接色值（如 `#6366F1`）应使用变量 |
| 组件合规 | 组件规格（按钮高度、圆角等）符合设计系统规范 |

#### 2.5 跨阶段回溯对齐

从当前阶段（设计规格）向前回溯到需求文档，验证端到端一致性。

**检查规则**：

| 检查项 | 说明 |
|--------|------|
| 字段名对齐 | 设计规格中的每个字段名是否与需求文档中的定义一致（如同一个字段不叫 `account_name` 又叫 `accountName`） |
| 枚举值对齐 | 状态、类型、平台等枚举值在两层文档中是否一致（如需求定义"监控中/已暂停"，设计规格不能变成"运行中/已停止"） |
| 操作有需求依据 | 设计规格中的每个操作能追溯到需求文档的用户故事或操作定义，无自行增加的操作 |
| 用户故事覆盖 | 每个用户故事在设计规格中有对应的页面/组件实现 |

**基准文件**：`开发文档/01-需求/*-需求.md`

## 与其他 skill 的关系

| Skill | 审查对象 | 阶段 |
|-------|---------|------|
| `/spec-review` | 设计规格文档 | 设计规格写完后 |
| `/design-review` | 页面描述文档 | 页面描述写完后 |
| `/check-page` | 前端实现 vs 设计稿 | 编码完成后 |
| `/code-review` | 代码质量 | 编码完成后 |

## 严重程度定义

| 标记 | 含义 | 示例 |
|------|------|------|
| ❌ 严重 | 缺失关键章节或字段 | 缺少响应式规则、需求字段未覆盖 |
| ⚠️ 警告 | 不完整但不影响主流程 | 子章节缺项、组件规格不够细 |
| 💡 建议 | 可优化项 | 硬编码值可替换为变量、描述可以更精确 |

## 注意事项

- 审查基于文档交叉比对，不涉及代码或设计稿文件
- 如果设计系统规范不存在，跳过2.4维度并标注
- 需求文档是功能的唯一基准，设计规格必须100%覆盖
- 设计系统规范是样式的唯一基准，不以其他设计规格为准
