# 需求审查 `/req-review`

> **使用场景**：`/refine` 输出需求文档后，检查需求质量。
> **目标**：确保需求文档无歧义、可测试、边缘场景充分，是下游所有阶段的可靠基准。
> **前置**：`/refine` 生成的需求文档
> **建议时机**：需求文档写完或修改后

## 执行方式（Subagent 模式）

本技能通过 Subagent 执行分析，不占用主对话上下文。

**必须使用 Agent tool** 启动分析代理，传入以下参数：
- subagent_type: "general-purpose"
- prompt: 包含下方完整审查清单

代理在独立上下文中读取需求文档，**只返回问题清单摘要给主对话**。

## 用法

```
/req-review                      # 审查所有需求文档
/req-review 自媒体内容生产系统     # 审查单个需求文档
```

## 执行步骤

### 第1步：收集文件（10秒）

扫描 `开发文档/01-需求/*-需求.md`，确定审查范围。

### 第2步：五维度审查（60秒）

#### 2.1 文档结构完整性

需求文档必须包含以下章节：

| # | 章节 | 检查内容 |
|---|------|---------|
| 1 | 一句话定义 | 格式：[谁] 需要 [什么] 来解决 [什么问题] |
| 2 | 用户故事 | 分类为核心/重要/锦上添花 |
| 3 | 页面清单 | 页面名称 + 对应用户故事 + 页面类型 |
| 4 | 逐页功能说明 | 每个页面6项（见下） |
| 5 | 页面导航关系 | 页面间跳转流程图 |
| 6 | 数据关系草图 | 核心实体关系 |
| 7 | 边缘场景 | 正常流 + 异常流 + 边界值 |
| 8 | 待确认问题 | 不确定项列表 |

**逐页功能说明6项检查**：

| 检查项 | 说明 |
|--------|------|
| 功能概述 | 每个页面有1句话功能描述 |
| 页面分区 | 每个页面有区块划分表 |
| 显示字段 | 每个区块有字段表（字段+显示方式+数据来源+备注） |
| 操作定义 | 每个区块有操作表（操作+触发方式+行为+权限/条件） |
| 页面状态 | 加载中/空数据/加载失败 三种状态 |
| 业务规则 | 页面级业务约束 |

#### 2.2 用户故事质量

**检查规则**：

| 检查项 | 说明 |
|--------|------|
| 格式正确 | "作为[角色]，我想要[做某事]，以便[达到某目的]" 三段式 |
| 核心故事数量 | 核心故事 ≤ 5个，超过需拆分 |
| 角色一致 | 故事中的角色在整个文档中定义明确 |
| 无实现细节 | 用户故事描述用户目标，不涉及技术方案 |
| 无重复故事 | 不同故事间不重叠 |
| 故事→页面覆盖 | 每个故事至少对应1个页面 |

#### 2.3 字段无歧义性

逐个检查每个页面的显示字段：

| 检查项 | 说明 |
|--------|------|
| 字段名唯一 | 同一页面内无重复字段名 |
| 数据来源明确 | 每个字段标注了来源（后端生成/用户输入/后端计算/关联查询） |
| 显示方式具体 | 不是"适当展示"，而是"文字加粗"、"标签颜色区分"等具体描述 |
| 格式说明 | 日期格式、金额格式、百分比格式等有说明 |
| 跨页面一致 | 同名字段在不同页面中的定义一致（如"状态"在各页面含义相同） |
| 无模糊词汇 | 不出现"类似XX"、"参考XX"、"适当"等模糊描述 |

#### 2.4 操作可测试性

逐个检查每个页面的操作：

| 检查项 | 说明 |
|--------|------|
| 触发方式明确 | 点击按钮/左滑/下拉/手势等具体触发方式 |
| 行为描述具体 | 操作后的结果可验证（跳转X页面/显示Y弹窗/调用Z接口） |
| 权限条件 | 操作的前置条件（状态限制/角色限制/数据限制） |
| 反馈描述 | 操作后的用户可见反馈（成功提示/失败提示/loading状态） |
| 操作→故事覆盖 | 每个操作可追溯到某个用户故事 |

#### 2.5 边缘场景充分性

| 检查项 | 说明 |
|--------|------|
| 正常流覆盖 | 每个核心操作有正常执行路径 |
| 异常流覆盖 | 网络错误、权限不足、数据冲突等异常路径 |
| 边界值覆盖 | 空数据、超长文本、数值为0、最大值等 |
| 状态组合 | 不同状态下操作的不同表现 |
| 并发场景 | 同一数据被多人同时操作 |
| 无"待定"场景 | 所有边缘场景都有明确的处理方式，不允许"待定" |

### 第3步：输出审查报告

```
📋 需求审查报告

审查范围：[N个需求文档]

### 汇总

| 维度 | ✅通过 | ⚠️警告 | ❌严重 |
|------|--------|--------|--------|
| 文档结构完整性 | N | N | N |
| 用户故事质量 | N | N | N |
| 字段无歧义性 | N | N | N |
| 操作可测试性 | N | N | N |
| 边缘场景充分性 | N | N | N |

### 文档结构问题

| # | 严重程度 | 页面 | 缺失项 | 说明 |
|---|---------|------|--------|------|
| 1 | ❌ | 全局 | 缺少"待确认问题"章节 | — |
| 2 | ⚠️ | 报价详情 | 缺少"业务规则" | — |

### 用户故事问题

| # | 严重程度 | 故事 | 问题描述 |
|---|---------|------|---------|
| 1 | ❌ | "系统自动推荐" | 无对应角色，格式不正确 |
| 2 | ⚠️ | 核心故事共7个 | 超过5个上限，建议拆分 |

### 字段歧义问题

| # | 严重程度 | 页面 | 字段 | 问题描述 |
|---|---------|------|------|---------|
| 1 | ❌ | 报价列表 | 金额 | 数据来源为"计算"但未说明计算公式 |
| 2 | ⚠️ | 报价详情 | 状态 | 显示方式为"标签"但未说明颜色映射规则 |

### 操作可测试问题

| # | 严重程度 | 页面 | 操作 | 问题描述 |
|---|---------|------|------|---------|
| 1 | ❌ | 新建报价 | 提交 | 未描述提交成功后的跳转行为 |
| 2 | ⚠️ | 报价列表 | 删除 | 未说明失败时的提示内容 |

### 边缘场景问题

| # | 严重程度 | 页面 | 场景 | 问题描述 |
|---|---------|------|------|---------|
| 1 | ❌ | 新建报价 | 价格为0 | 未处理 |
| 2 | ⚠️ | 报价列表 | 并发删除 | 未考虑 |

### 修复建议
1. [具体修复方案]
2. [具体修复方案]
```

## 与其他审查技能的关系

| Skill | 审查对象 | 阶段 |
|-------|---------|------|
| `/project-review` | 项目配置 | 项目初始化后 |
| `/req-review` | 需求文档 | `/refine` 后 |
| `/spec-review` | 设计规格 | `/design-spec` 后 |
| `/design-review` | 页面描述 | `/design-detail` 后 |
| `/check-page` | 前端实现 vs 设计稿 | `/design-page` 后 |
| `/dataflow-review` | 数据流文档 | `/data-flow` 后 |
| `/techspec-review` | 技术规格 | `/tech-spec` 后 |
| `/code-review` | 代码质量 | 编码后 |

## 严重程度定义

| 标记 | 含义 | 示例 |
|------|------|------|
| ❌ 严重 | 会导致下游阶段产生歧义或遗漏 | 字段无数据来源、操作无触发方式、核心故事超5个 |
| ⚠️ 警告 | 不影响主流程但不够规范 | 格式描述模糊、显示方式不具体、缺少非核心边缘场景 |
| 💡 建议 | 可优化项 | 用户故事表述可更精炼、业务规则可更详细 |

## 注意事项

- 需求文档是整个流水线的源头，问题越早发现修复成本越低
- 需求审查只关注"要做什么"，不评判"怎么做"（技术方案由下游审查）
- 如果需求文档中的某个页面缺少功能说明，标记为严重——下游所有阶段都依赖它
- "待确认"问题本身不算缺陷，但如果超过5个说明需求尚未成熟
