# 新功能开发工作流 `/wf-new`

> **使用场景**：开发一个新功能、新页面、新模块时。
> **目标**：一条命令走完全流程，从需求确认到生产部署。

## 用法

```
/wf-new <需求描述>                    # 一条龙（自动对齐+计划+开发+检查+部署）
/wf-new admin新增财务管理报表            # 带具体描述
/wf-new 客户端增加订单导出PDF功能      # 带具体描述
/wf-new --plan-only                     # 只生成计划不执行（评审用）
```

---

## 流程图

```
/wf-new:
  align → plan → [自动执行计划] → check-page → perf-check → component-check → deploy-check → bushu
```

---

## 步骤1：需求对齐（强制，不可跳过）

**技能**: `/align` 自动执行

**每次都必须执行**，通过问答明确需求后再进入计划阶段。

执行流程：
1. 读取用户的需求描述作为初始输入
2. 执行 `/align` 7问对齐，逐项确认：
   - 功能边界（做什么/不做什么）
   - 涉及模块（哪些页面/组件）
   - 数据来源（哪些表/接口）
   - 展示要求（列表/表单/弹窗）
   - 业务规则（校验/权限/状态流转）
   - 优先级与约束（必须/可选/截止时间）
   - 验收标准（怎样算完成）
3. 对齐结果确认后，进入步骤2

**原则**: 宁可多问一句，不要返工一次。

---

## 步骤2：制定实施计划

**技能**: `/plan` 自动生成

根据对齐结果（或用户需求），生成结构化的实施计划：

### 计划模板

```markdown
## 实施计划：<功能名称>

### 阶段一：数据层（后端）
- [开发] 创建XXX Model
- [开发] 创建XXX Schema
- [开发] 创建XXX API端点
- [开发] 创建XXX Service

### 阶段二：前端层
- [开发] 创建/修改 XXX 页面组件
- [开发] 添加前端API调用函数
- [开发] 更新路由配置（如需要）

### 阶段三：验证与部署
- [页面] 对照设计稿检查（/check-page）
- [性能] 性能预算检查（/perf-check）
- [组件] 组件统一性检查（/component-check）
- [检查] 部署前检查（/deploy-check）
- [部署] 部署到生产（/bushu）
```

### 自动分析规则

生成计划时自动分析：

| 分析维度 | 操作 |
|---------|------|
| 涉及文件数 | > 3 个文件时分阶段 |
| 是否有新数据库表 | 有 → 先建 Migration |
| 是否有新API | 有 → 标注 Schema + Endpoint |
| 是否有前端页面 | 有 → 标注对照设计稿 |
| 是否涉及财务/权限 | 是 → 加 `/scan-algo` 审查 |
| 是否涉及调度管道 | 是 → 加 `/scan-algo` 审查 |

**输出**: 结构化计划文本，展示给用户确认。

---

## 步骤3：执行计划

**引擎**: `/wf-batch` 逐个执行计划中的任务

执行规则：
- **顺序执行**：按计划顺序逐个任务，前面的结果影响后面的
- **不中断**：遇到无法自动决定的事项，记录到阻塞列表，跳过继续
- **自动判断**：每个任务根据类型自动匹配执行策略（同 `/wf-batch`）
- **提交策略**：每个阶段完成后自动 `git commit`，不等到最后一起提交

### 任务类型自动映射

| 计划标记 | 执行策略 | 说明 |
|---------|---------|------|
| `[开发]` | 直接开发 | 读取设计稿+数据文档，编写代码 |
| `[审查]` | `/code-review` | 代码质量审查 |
| `[审查+修复]` | `/wf-review` | 审查并修复 |
| `[算法]` | `/wf-algo` | 调度管道审查+修复 |
| `[页面]` | `/check-page` | 对照设计稿检查 |
| `[性能]` | `/perf-check` | 性能预算检查 |
| `[组件]` | `/component-check` | 组件统一性检查 |
| `[检查]` | `/deploy-check` | 部署前检查 |
| `[部署]` | `/bushu` | 部署到生产 |

---

## 步骤4：自动插入验证步骤

计划执行器在以下位置自动插入验证步骤，不需要用户手动写：

| 插入时机 | 自动添加的步骤 |
|---------|--------------|
| 计划中有 `[页面]` 任务 | `[页面]` 后自动加 `[性能]`（如果是列表页） |
| 所有开发任务完成后 | 自动加 `[组件]` → `[检查]` → `[部署]` |
| 计划中有 `[部署]` | 正常执行 |
| 计划中没有 `[部署]` | 最后自动加 `[检查]`（开发模式，不部署） |

## 与现有工作流的关系

| 工作流 | 关系 |
|-------|------|
| `/wf-new` | **编排器**，串联全流程 |
| `/wf-dev` | `/wf-new` 的子集（已跳过计划和自动执行） |
| `/wf-batch` | `/wf-new` 的执行引擎（步骤3实际调用） |
| `/wf-review` | 算法/审查类任务自动路由到此 |
| `/wf-bug` | bug修复类任务自动路由到此 |
| `/wf-deploy` | 部署阶段自动路由到此 |
| `/align` | 需求阶段自动调用（可跳过） |
| `/plan` | 计划阶段自动调用 |
| `/save-ctx` | 部署完成后自动保存上下文 |

## 示例

### 示例1：新功能开发

```
用户: /wf-new 做一个财务看板

自动执行流程：
1. 执行 /align → 7问对齐（哪个模块？哪些指标？展示形式？）
2. 根据对齐结果生成计划
3. 用户确认计划
4. 逐个执行
```

### 示例2：明确需求也能对齐

```
用户: /wf-new 在client端报价页面添加AI智能报价建议按钮

自动执行流程：
1. 执行 /align → 快速确认（AI建议的数据来源？触发时机？展示形式？）
2. 生成计划 →
   - [开发] 创建后端AI分析服务
   - [开发] 添加AI建议API端点
   - [修改] CreateQuotationDialog.vue 添加AI建议按钮
   - [开发] 前端API调用
   - [页面] 对照设计稿检查
   - [检查] → [部署]
3. 逐个执行
```

## 原则

- **需求优先**：先确认做什么，再想怎么做
- **双源对照**：前端开发必须对照 shejigao.pen + 数据逻辑文档
- **分阶段提交**：每个阶段完成后自动 commit，降低回滚风险
- **不中断原则**：遇到阻塞记录跳过，不等待确认（同 `/wf-batch`）
- **验证不跳过**：check-page、perf-check、deploy-check 不可跳过
- **自动部署决策**：根据计划中是否有"部署"关键词决定是否部署
