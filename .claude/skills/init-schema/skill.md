# 初始化共享定义 `/init-schema`

> **使用场景**：`/refine` 完成需求文档后、`/design-spec` 之前，第一时间建立命名锚点。
> **目标**：将需求文档中的枚举、实体名、字段命名提取为 `shared-schema.ts`，后续阶段只引用不创造。
> **前置**：`/refine` 生成的需求文档
> **后置**：`/design-spec`（引用 schema 中的命名）

## 用法

```
/init-schema                    # 从需求文档提取共享定义
```

## 执行步骤

### 第1步：读取需求文档

扫描 `开发文档/01-需求/*-需求.md`，提取：

| 提取项 | 来源位置 | 示例 |
|--------|---------|------|
| 枚举类型 | 页面功能说明中的状态、类型、平台 | 任务状态、素材类型、平台名 |
| 实体名称 | 数据关系草图 | 用户、任务、素材、脚本、视频 |
| 字段命名 | 显示字段表 | 每个字段的英文代码名 |
| 工作流步骤 | 用户故事中的流程 | 采集→选题→制作→发布→监控 |

### 第2步：建立命名约定

从需求文档的表述中推导出统一的命名规则：

```typescript
// 命名规则
// 1. 代码标识符使用英文小写+下划线（snake_case 用于 DB/API，camelCase 用于 TS）
// 2. 枚举值使用英文小写（前端通过 Label Map 显示中文）
// 3. 状态值使用进行时/过去分词（如 'collecting' | 'completed'）
// 4. 平台/类型使用简短英文标识（如 'xiaohongshu' | 'douyin' | 'weixin'）
```

### 第3步：生成 shared-schema.ts

在 `开发文档/06-架构/` 目录下创建 `shared-schema.ts`，初始版本包含：

```typescript

 * shared-schema.ts — 唯一真相源
 * 所有字段名、枚举、API端点、表结构只在此定义。
 * 后续阶段（/data-flow, /tech-spec）逐步扩展。
 */

// ============================================================

// 每个枚举包含：
// - Type 类型别名（英文值）
// - LABELS 常量（中文映射）

export type Platform = 'xiaohongshu' | 'douyin' | 'weixin'
export const PLATFORM_LABELS: Record<Platform, string> = { ... }

// ... 从需求文档提取的所有枚举

// ============================================================
// §2 核心数据结构
// ============================================================

// 从需求文档的数据关系草图中提取的核心实体结构

// ============================================================
// §3 数据库表结构（由 /tech-spec 阶段扩展）
// ============================================================

// ============================================================
// §4 API 端点常量（由 /data-flow 阶段扩展）
// ============================================================

// ============================================================
// §5 路由配置
// ============================================================

// ============================================================
// §6 跨页面参数（由 /data-flow 阶段扩展）
// ============================================================
```

### 第4步：验证命名覆盖

将需求文档中的每个中文术语与 schema 中的英文命名对照：

| 中文术语 | 英文命名 | 枚举/类型 |
|---------|---------|---------|
| 小红书 | xiaohongshu | Platform |
| 抖音 | douyin | Platform |
| 采集 | collecting | TaskStatus |
| ... | ... | ... |

输出确认：
```
✅ shared-schema.ts 已创建：开发文档/06-架构/shared-schema.ts
📊 初始版本：[N]个枚举 + [M]个核心实体 + 命名约定
→ 下一步：/design-spec（引用 schema 命名，不允许自行创造）
```

## 内含规则（强制）

1. **命名只做一次**：所有枚举值、实体名、字段名只在此步骤定义，后续阶段只能引用
2. **英文+中文映射**：代码标识符使用英文，通过 `LABELS` 常量映射中文显示
3. **新字段必须先注册**：如果后续阶段发现 schema 遗漏的字段，必须先在 schema 中注册再使用
4. **分阶段扩展**：初始版本只有枚举+实体名，API/表结构由后续阶段补充
5. **审查必检**：所有审查技能（/spec-review, /design-review 等）必须检查命名与 schema 一致

## 与其他技能的关系

```
/refine → /init-schema → /design-spec → /design-detail → /design-page
              ↓ 锚定         ↓ 引用         ↓ 引用         ↓ 引用
         /data-flow → /tech-spec → 编码
              ↓ 扩展§4-§6   ↓ 扩展§3     ↓ import直接用
```

- `/refine` 产出的需求文档是本技能的输入
- 本技能产出的 `shared-schema.ts` 是所有下游阶段的命名基准
- `/data-flow` 完成后扩展 schema 的 §4（API端点）和 §6（跨页面参数）
- `/tech-spec` 完成后扩展 schema 的 §3（数据库表结构）
- 编码阶段直接 `import` 使用，不翻文档

## 审查技能的 schema 一致性检查

以下审查技能需增加 schema 一致性检查维度：

| 审查技能 | 检查内容 |
|---------|---------|
| `/spec-review` | 设计规格引用的枚举值与 schema §1 一致 |
| `/design-review` | 页面描述的字段名与 schema §2 一致 |
| `/dataflow-review` | 数据流的字段名、API端点与 schema 一致 |
| `/techspec-review` | 技术规格的类型定义、表结构与 schema 一致 |
