# 全自动流水线 `/wf-full`

> **使用场景**：需求文档已完善（通过 `/refine`），一键跑通从设计到部署的全链路。
> **核心机制**：每步自动执行 + 自动审查，审查通过则自动进入下一步，审查失败则暂停展示问题。
> **与 /wf-new 的区别**：`/wf-new` 从对齐开始到编码部署；`/wf-full` 从设计规格开始，覆盖更长的链路。

## 用法

```
/wf-full                                    # 交互式，选择要处理的页面
/wf-full 全部                               # 批量处理所有未完成的页面
/wf-full 选题工作台                          # 单页面流水线
/wf-full 选题工作台 素材库 订单池             # 多页面流水线
/wf-full --resume                           # 从上次暂停处继续
```

## 执行规则

### 通用规则

1. **按页面逐个处理**：每个页面走完 ①→⑧ 全链路后，再处理下一个页面
2. **审查不跳过**：每步产出后必须运行对应审查，审查通过才进入下一步
3. **失败暂停**：审查发现问题 → 用中文展示问题清单 → 等待用户指示（"继续" / "修复后继续"）
4. **自动提交**：每个步骤完成后自动 `git commit`
5. **进度追踪**：在 `.claude/wf-full-progress.md` 中记录当前状态

### 审查执行方式

所有审查使用 Subagent 模式（不占主窗口）：

```
Agent(subagent_type: "general-purpose", prompt: "执行 {审查技能名} 的完整审查清单...")
```

审查代理只返回问题清单摘要给主对话。无问题 → 标记"通过"；有问题 → 暂停。

## 步骤详解

### 步骤 ① 设计规格 `/design-spec`

**读取**：`/refine` 输出的需求文档
**产出**：`开发文档/02-设计/{页面名}-设计规格.md`

执行方法：读取 `design-spec` 技能卡的完整流程，按步骤执行。

**关键动作**：
1. 读取需求文档，确认页面名称和类型
2. 按7要素（颜色/字体/间距/组件/状态/响应式/交互）生成设计规格
3. 确保与设计系统规范一致

### 步骤 ①R 设计规格审查 `/spec-review`

**审查内容**：7要素完整性 + 需求覆盖度 + 设计系统一致性
**通过标准**：无阻塞问题（⚠️ 警告可继续，❌ 阻塞必须修复）

### 步骤 ② 页面描述 `/design-detail`

**读取**：步骤①产出的设计规格
**产出**：`开发文档/02-设计/{页面名}-页面描述.md`

执行方法：读取 `design-detail` 技能卡的完整流程。

**关键动作**：
1. 读取设计规格，提取7要素参数
2. 生成逐区块详细描述（元素/文案/尺寸/数据绑定/操作/状态）
3. 标注字段溯源（📥API获取 🔗关联表 📊计算 ⛔常量）

### 步骤 ②R 页面描述审查 `/design-review`

**审查内容**：跨页面一致性 + 样式变量合法性 + 功能映射完整性
**通过标准**：无阻塞问题

### 步骤 ③ 视觉稿 `/design-page`

**读取**：步骤②产出的页面描述 + 设计规格
**产出**：shejigao.pen 中的新页面

执行方法：读取 `design-page` 技能卡的完整流程。

**关键动作**：
1. 读取页面描述和设计规格
2. 获取设计系统变量（59个）
3. 在 Pencil 中生成页面视觉稿
4. 截图确认视觉效果

**注意**：此步骤不做独立审查，视觉质量在步骤③ data-flow 时通过截图间接验证。

### 步骤 ④ 数据流设计 `/data-flow`

**读取**：步骤③产出的设计稿 + 页面描述
**产出**：`开发文档/04-数据流/{页面名}-数据流.md`

执行方法：读取 `data-flow` 技能卡的完整流程。

**关键动作**：
1. 从设计稿提取所有字段
2. 字段溯源（来源/类型/校验规则/默认值）
3. API 契约设计（端点/方法/请求体/响应体）
4. 状态管理设计（store 字段/异步 action）
5. **完成后自动扩展 `shared-schema.ts`**（新 API 端点、新枚举值）

### 步骤 ④R 数据流审查 `/dataflow-review`

**审查内容**：字段溯源完整性 + API 跨页面一致性 + 与页面描述映射完整性
**通过标准**：无阻塞问题

### 步骤 ⑤ 技术规格 `/tech-spec`

**读取**：步骤④产出的数据流 + 页面描述 + 设计稿
**产出**：`开发文档/05-技术规格/{页面名}-技术规格.md`

执行方法：读取 `tech-spec` 技能卡的完整流程。

**关键动作**：
1. 生成组件层级树
2. 定义每个组件的 Props/Events/Slots
3. API 调用规格（端点/参数/错误处理）
4. 表结构变更（如需）
5. **完成后自动扩展 `shared-schema.ts`**（新路由、新组件类型）

### 步骤 ⑤R 技术规格审查 `/techspec-review`

**审查内容**：字段/API/类型与数据流对齐 + TypeScript 类型完整性 + 组件层级合理性
**通过标准**：无阻塞问题

### 步骤 ⑥ 施工图 + 自动施工 `/plan-first`

**读取**：步骤⑤产出的技术规格
**产出**：PLAN.md（施工图）+ 完成的代码

执行方法：读取 `plan-first` 技能卡的完整流程。

**关键动作**：
1. Phase 1：依赖分析 → 生成施工图（PLAN.md）
2. **STOP**：展示施工图，等待用户 `[Approved]`
3. Phase 2：自动施工（逐任务执行 → 验证完成标准 → 自动修复 → 暂停汇报）
4. 每阶段完成后自动 `git commit`

### 步骤 ⑦ 最终验证 `/check-page`

**审查内容**：页面实现与设计稿 + 数据流文档的一致性
**通过标准**：无阻塞性差异

执行方法：读取 `check-page` 技能卡的完整流程。

### 步骤 ⑧ 部署 `/bushu`

**前置**：步骤⑦ 通过
**执行**：读取 `bushu` 技能卡，执行部署流程

## 暂停与恢复

### 暂停条件（仅以下情况暂停）

| 条件 | 处理 |
|------|------|
| 审查发现 ❌ 阻塞问题 | 展示问题清单，等待用户确认 |
| plan-first 需要用户确认施工图 | 展示 PLAN.md，等待 `[Approved]` |
| 自动修复 3 次仍失败 | 汇报问题，等待用户指示 |
| 遇到业务决策点 | 用中文说明选项，等待选择 |

### 恢复

用户说以下任一关键词即恢复执行：
- "继续" / "修复后继续" — 从暂停步骤继续
- `/wf-full --resume` — 从进度文件记录的位置继续

## 批量模式

```
/wf-full 全部
```

执行逻辑：
1. 扫描 `开发文档/01-需求/` 中的需求文档
2. 对比 `开发文档/02-设计/` 中已有的设计规格，确定未完成的页面
3. 逐页面执行 ①→⑧ 全链路
4. 每完成一个页面，输出简要进度报告

**汇报格式**：

```
📋 /wf-full 批量进度

[3/7] 素材库 ✅ 完成
[4/7] 选题工作台 ⏳ 步骤④ 数据流设计中
[5/7] 订单池 ⏸️ 等待确认（审查发现2个问题）
[6/7] 待处理
[7/7] 待处理
```

## 与现有工作流的关系

| 工作流 | 覆盖范围 | 使用场景 |
|--------|---------|---------|
| `/wf-full` | 需求→设计→编码→部署（全链路） | 需求已完善，一键出结果 |
| `/wf-new` | 对齐→编码→部署 | 已有技术规格，快速编码 |
| `/wf-dev` | 手动开发→检查 | 没有技术规格的快速迭代 |
| `/plan` + `/wf-batch` | 批量多任务 | 大量独立小任务 |

---

## 原则

1. **审查是保险丝，不是路障**：审查发现问题是为了防止错误累积，不是为了增加步骤
2. **自动推进是默认行为**：不需要用户说"下一步"，系统自己推进
3. **中文沟通**：所有暂停时的问题描述使用中文
4. **可中断可恢复**：任何时候可以中断，`--resume` 从断点继续
5. **进度透明**：用户随时能看到当前执行到哪一步
